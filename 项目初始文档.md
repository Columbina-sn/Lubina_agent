## 1. 竞品分析：RemindAI 深度拆解

### 1.1 项目概览

| 维度 | 详情 |
|------|------|
| **作者** | PythonnotJava（独立开发者） |
| **仓库** | https://github.com/PythonnotJava/RemindAI |
| **官网** | https://www.robot-shadow.cn/extend/remindai/ |
| **技术栈** | Flutter（Dart） |
| **定位** | 开源桌面 AI 助手，不止于聊天——是一个带工具调用能力的 Agent |
| **许可证** | 开源 |

### 1.2 核心功能拆解

#### (1) 文件系统操作
- 支持文件的**读、写、删、改**四种原子操作
- Agent 可以真正"操作"用户的本地文件，而不只是生成文本
- 这是从"聊天 bot"跨越到"工具 agent"的关键一步

#### (2) Skill 技能系统 + MCP 协议全覆盖
这是 RemindAI 最核心的设计亮点：

- **三种 MCP 传输方式均已支持**：
  - Stdio（标准输入输出，适用于本地进程通信）
  - SSE（Server-Sent Events，适用于服务端推送）
  - Streamable HTTP（适用于 Web 服务集成）
- **可扩展 Skill 机制**：用户可以自行编写和安装技能模块
- 这意味着 RemindAI 的 Agent 理论上可以调用任何符合 MCP 协议的外部工具

> **MCP 协议**（Model Context Protocol）是 Anthropic 提出的 AI 工具调用标准协议。简单理解就是：AI 和工具之间的一种"通用语言"。支持 MCP 意味着第三方开发者可以无穷无尽地给这个 Agent 添加新能力。

#### (3) 向量记忆——跨会话不遗忘
- 基于向量数据库的长期记忆系统
- 对话内容在会话结束后**不会丢失**
- 下一次对话时，Agent 能回忆起之前的上下文
- 配套有一个 npm 包 `@remind_ai/remind-mcp`，提供了 `remember`、`recall`、`consolidate`、`forget` 四个核心 API

这是一个非常有价值的特性——普通聊天 AI 每次对话都是"失忆"的，而 RemindAI 能记住用户是谁、在做什么项目、有什么偏好。

#### (4) 多 Agent 协作
- 支持多个 Agent 实例协同工作
- 可以模拟"一个 Agent 做规划，另一个 Agent 做执行"的分工模式
- 通过 MCP 协议在 Agent 之间共享上下文

#### (5) 多模型原生适配
- **OpenAI**（GPT 系列）
- **Claude**（Anthropic 系列）
- **Gemini**（Google 系列）
- 用户可以自由切换，不绑定单一供应商

### 1.4 RemindAI 的优点

| 优点 | 详细说明 |
|------|---------|
| **MCP 协议支持深入** | 三种传输方式全支持，拓展性极强，这是市面上绝大多数同类产品做不到的 |
| **记忆系统闭环** | 从记忆存储→检索→整合→遗忘，形成完整的记忆生命周期 |
| **技术栈统一** | 全 Flutter/Dart，无跨语言调用开销，打包分发简单 |
| **多模型灵活** | 不锁定单一供应商，用户可根据需求/预算切换 |
| **开源且可自托管** | API Key 由用户自己提供，无服务器成本压力 |

### 1.5 RemindAI 的不足与局限

| 不足 | 详细说明 | 对我们的启示 |
|------|---------|------------|
| **桌面能力弱** | Flutter 的桌面支持（特别是 Windows）仍不够成熟：系统托盘、全局快捷键、开机自启、文件拖拽等原生功能实现成本高 | 我们选择 Tauri 的核心优势就在这里——原生系统集成 + 轻量 |
| **目标用户模糊** | 没有明确的用户画像——程序员觉得不如 Cursor/Copilot 专业，普通用户觉得太技术化 | 我们必须明确：**面向大学生，场景驱动，非工具驱动** |
| **缺少文档/Office 集成** | 不支持直接操作 Word/PPT/Excel，这恰好是大学生最高频的需求 | 这是我们的核心差异化方向 |
| **缺少习题/学习场景** | 没有针对学习场景的专项优化 | 错题整理、习题解析是我们的特色功能 |
| **多模态缺失** | 不支持生图、生视频、图片理解等多模态能力 | 可作为进阶功能加入 |
| **单人项目持续维护难** | 开源后依靠社区贡献，功能和稳定性迭代不稳定 | 你的项目起步阶段也是如此，需要控制好节奏和预期 |

## 2. RemindAI 对我们的启发与差异化空间

### 2.1 关键启发

1. **MCP 协议是正确方向**：RemindAI 证明了 MCP 协议在桌面 Agent 中是可行且高效的。我们应该从一开始就设计 MCP 兼容的工具调用框架。Claude Code、Cursor 都在用 MCP，这是行业共识。

2. **记忆系统是杀手特性**：跨会话记忆让用户产生"AI 认识我"的感觉，极大提升粘性。RemindAI 的 `remember → recall → consolidate → forget` 四步记忆生命周期设计非常值得借鉴。

3. **开源 + 自带 Key 的模式可行**：RemindAI 不承担 API 成本，用户自己申请 Key。这与你设想的"少数人自己配 Key，多数人买月卡"的模式高度一致。

4. **Skill 可扩展性是护城河**：让用户/社区可以自建 Skill，产品的生命力就不再取决于开发者一个人的时间。长期来看，这是对抗商业竞品的关键。

5. **Flutter 做桌面有天花板**：从 RemindAI 选择 Flutter 来看，它的跨平台能力好，但对于需要深度系统集成的桌面应用（托盘、快捷键、文件拖拽、进程守护），Flutter 的能力有限。这印证了我们选择原生壳方案（Tauri）的合理性——比 Flutter 更贴近系统，比 CEF 轻量得多。

### 2.2 我们的差异化空间

| 竞争维度 | RemindAI | Claude Code / Cursor | **[项目名待定] AI（我们）** |
|---------|----------|---------------------|---------------------|
| 目标用户 | 泛开发者 | 专业程序员 | **大学生（本科/研究生）** |
| 核心场景 | 通用 AI 助手 | 代码编写 | **学习 + 文档 + 代码入门** |
| 桌面体验 | 一般（Flutter） | 优秀（原生） | **优秀（Tauri 原生）** |
| Office 集成 | ❌ | ❌ | ✅ **核心差异化** |
| 习题/错题 | ❌ | ❌ | ✅ **核心差异化** |
| 多模态 | 有限 | 有限 | ✅ **生图/生视频/图片理解** |
| MCP 支持 | ✅ 优秀 | ✅ 优秀 | ✅ **必须支持** |
| 价格 | 免费（自带 Key） | $20/月 | **免费为主 + 低成本月卡** |
| 上手门槛 | 中等 | 高 | **低（场景引导，非工具引导）** |

### 2.3 核心差异化策略总结

> **一句话**: 做一个"大学生开箱即用"的桌面 Agent——不需要懂技术、不需要配环境、不需要看文档，打开就能让 AI 帮你写论文、整笔记、做 PPT、查错题。

## 3. 我们的产品定位

### 3.1 用户画像

**主要用户**：大一 ~ 研三的在校学生，满足以下特征：
- 有基本的电脑操作能力（会用 Word、浏览器）
- 有学业任务需要完成（论文、报告、PPT、习题）
- **不一定懂编程，不一定知道什么是 API Key**
- 对 AI 有好奇心但不了解具体能做什么
- 愿意尝试免费/低价工具来提高效率

**次要用户**：对 AI 有一定了解的学生，愿意自己配置 API Key 使用更强大的模型。

### 3.2 产品一句话

> [项目名待定] AI —— 一个打开就能用的桌面学习伙伴。告诉它你要写什么作业，它帮你搞定。

### 3.3 核心功能优先级

| 功能 | 说明 | 复杂度 |

### 4.2 各层技术选型详解

#### 4.2.1 桌面外壳层：Tauri 2.0（推荐方案）

> ⚠️ **重要更新（2026-06-27）**：原方案 C++/CEF 经最新调研后**不再推荐**。以下说明原因并给出新方案。

##### 为什么不用 C++/CEF 了？

| 维度 | CEF 实际数据 | 对大学生用户的影响 |
|------|------------|------------------|
| **安装包体积** | ~100-150MB（捆绑完整 Chromium） | 宿舍网速差，下载半天 → 用户流失 |
| **空闲内存** | ~150MB（多进程架构：Browser + Renderer + GPU） | 8GB 笔记本跑几个 Chrome 标签页就爆了 |
| **开发效率** | C++ 开发+调试 ≈ Python 的 1/3~1/5，CEF 生命周期管理极复杂 | 阶段 0 预研就可能耗 2~3 个月 |
| **实际代码量** | 不是 8-10%，CEF 的 IPC/生命周期/跨进程消息处理至少 20-30% | 大量精力花在和产品无关的框架胶水上 |
| **更新维护** | 需手动跟踪 CEF ↔ Chromium 版本对应 | 自找麻烦 |

C++/CEF 适合的场景：Steam 客户端、Spotify、Epic Games Launcher——这些是"已有大型 C++ 原生应用，需要嵌入一个浏览器"。你的项目是从零开始的桌面 Agent，不是这个场景。

##### 为什么 Cursor 用 Electron 而你不需要？

Cursor 是 **fork 了 VS Code 的开源代码**。VS Code 本身就是 Electron 写的，Cursor 继承了这个壳，省下了"造编辑器"的巨量工作。这是继承，不是选择。

你的项目是从零开始。Electron 的空壳就要 150MB+ 安装包、200MB 空闲内存——对学生笔记本不友好。

##### 新方案：Tauri 2.0

| 维度 | Tauri 2.0 | Electron（Cursor） | CEF（旧方案） |
|------|-----------|-------------------|-------------|
| **安装包** | **3-5 MB** | 150+ MB | 100-150 MB |
| **空闲内存** | **30-50 MB** | 150-300 MB | ~150 MB |

**你需要写的 Rust 代码（全部，约 250 行）**：
```rust
// src-tauri/src/main.rs —— 整个桌面壳的代码

#![cfg_attr(not(debug_assertions), windows_subsystem = "windows")]

use tauri::{
    CustomMenuItem, Manager, SystemTray, SystemTrayEvent, SystemTrayMenu,
    GlobalShortcutManager,
};
use std::process::{Child, Command};
use std::sync::Mutex;

struct PythonBackend(Mutex<Option<Child>>);

fn main() {
    // 1. 启动 Python 后端
    let python_process = Command::new("./python_runtime/backend.exe")
        .spawn()
        .expect("无法启动 Python 后端");

    tauri::Builder::default()
        // 2. 管理 Python 进程生命周期
        .manage(PythonBackend(Mutex::new(Some(python_process))))
        .on_window_event(|event| {
            // Python 后端进程守护：崩溃自动重启
        })
        // 3. 系统托盘
        .system_tray(SystemTray::new().with_menu(
            SystemTrayMenu::new()
                .add_item(CustomMenuItem::new("show", "显示窗口"))
                .add_item(CustomMenuItem::new("quit", "退出"))
        ))
        .on_system_tray_event(|app, event| match event {
            SystemTrayEvent::MenuItemClick { id, .. } => match id.as_str() {
                "show" => { app.get_window("main").unwrap().show().unwrap(); }
                "quit" => {
                    // 先停 Python，再退出
                    let state = app.state::<PythonBackend>();
                    if let Some(mut child) = state.0.lock().unwrap().take() {
                        child.kill().ok();
                    }
                    app.exit(0);
                }
                _ => {}
            },
            _ => {}
        })
        // 4. 注册全局快捷键
        .setup(|app| {
            app.global_shortcut_manager()
                .register("Alt+Space")
                .unwrap();
            Ok(())
        })
        // 5. 加载前端
        .run(tauri::generate_context!())
        .expect("启动失败");
}
```

> **Rust 恐惧症？** 上面这段就是你**全部需要写的 Rust**。其余 100% 的代码还是 Python + HTML/CSS/JS。Tauri 的 Rust 层只做一件事——替代原来计划的 C++/CEF 做窗口壳。200 行 Rust 的学习成本远低于 3000 行 C++/CEF。

##### 关于 "Win32" 的澄清

你在文档里看到 "Win32 窗口编程"，那个词不是指 32 位系统。"Win32 API" 是 Windows 编程接口的历史名称，Win11 x64 照样用这套 API。就像 "C++" 不叫 "C64" 一样——名字和历史有关，和技术代际无关。换成 Tauri 方案后，你不需要直接接触 Win32 API，Tauri 帮你封装好了。

##### 备选方案：如果你执意用 C++

如果 C++ 是你的学习目标之一（而不仅仅是项目工具），可以考虑 **Microsoft WebView2 + C++** 替代 CEF：

| 对比 | WebView2（备选） | CEF（不推荐） |
|------|-----------------|-------------|
| 浏览器引擎 | 系统 Edge Chromium（共享） | 捆绑私有 Chromium |
| 安装包增量 | ~2MB（只含 SDK dll） | ~100MB |
| API 风格 | COM 接口（和 Win32 一样古老但标准） | C++ 类封装 |
| 官方支持 | Microsoft 官方维护，文档齐全 | 社区 + Spotify/Google 工程师维护 |
| 更新 | 随 Edge 自动更新 | 手动跟踪版本 |

但这仍然比 Tauri 方案复杂得多。**建议**：项目用 Tauri 快速出成品，C++ 学习可以另找一个纯 C++ 的练手项目（比如写个简单的游戏引擎 Demo）。

### 4.3 Tauri 外壳 ↔ Python 后端的通信方案

架构本质没变——Python FastAPI 跑在本地，Tauri 负责进程管理和窗口渲染。

```
┌─────────────────────────────────────────────┐
│              Tauri 桌面壳 (Rust)             │
│  ┌──────────┐ ┌──────────┐ ┌────────────┐  │
│  │ 窗口管理  │ │ 系统托盘  │ │ 快捷键注册  │  │
│  │ (Win/Mac │ │          │ │ Alt+Space  │  │
│  │  /Linux) │ │          │ │            │  │
│  └────┬─────┘ └────┬─────┘ └─────┬──────┘  │
│       │             │             │         │
│  ┌────▼─────────────▼─────────────▼──────┐  │
│  │          进程守护                      │  │
│  │  启动 Python → 健康检查 → 崩溃重启     │  │
│  └────────────────┬──────────────────────┘  │
│                   │                         │
│  ┌────────────────▼──────────────────────┐  │
│  │        WebView（Edge Chromium）        │  │
│  │        加载前端 HTML/CSS/JS            │  │
│  └────────────────┬──────────────────────┘  │
│                   │                         │
│         ══════════│══════════                │
│         ══ HTTP ══│══════════  前端通过      │
│         ══════════│══════════  fetch 调后端  │
│                   │                         │
├───────────────────┼─────────────────────────┤
│         业务核心层│  Python 3.11 + FastAPI   │
│                   │  localhost:19800         │
└───────────────────┴─────────────────────────┘
```

**Tauri 启动流程**：
```
1. 用户双击 exe
2. Tauri (Rust) 启动 → 创建主窗口
3. Rust 通过 std::process::Command 启动 Python 子进程
4. 轮询 localhost:19800/health 等待后端就绪（通常 <2 秒）
5. WebView 加载前端 index.html
6. 前端 JS 通过 fetch 连接 localhost:19800
7. 用户看到聊天界面 → 可以开始使用
```

**Tauri 退出流程**：
```
1. 用户点关闭 / 右键托盘"退出"
2. Tauri 发送 POST localhost:19800/shutdown
3. 等待 Python 进程优雅退出（超时 3 秒）
4. 超时则 kill 进程
5. Tauri 销毁窗口，释放资源
```

**与旧方案（C++/CEF）的对比**：

| 通信维度 | Tauri 方案 | CEF 方案 |
|---------|-----------|---------|
| 前端 → Python | `fetch("http://localhost:19800/...")` | 同左 |
| 前端 → 原生能力 | `@tauri-apps/api` JS 包（如 `openFileDialog()`） | CEF JS Bridge + C++ 代码 |
| Python 进程管理 | Rust `std::process::Command`（10 行） | C++ `CreateProcess` + 消息循环（~100 行） |
| 进程守护 | Rust 代码，简单 | C++ 代码，复杂 |
| 崩溃恢复 | Rust watch 子进程状态 | C++ WaitForSingleObject |

**核心优势**：前端调 Python 的方式完全不变（就是 HTTP fetch）。唯一变的是原生能力（文件对话框等）通过 `@tauri-apps/api` 而非自己写 C++ 桥接代码。

#### 4.3.3 SQLite 迁移：桌面 App 特有的数据库升级地狱

> Web 应用跑在服务器上，升级是一瞬间的事。桌面 App 用户分散在千千万万台电脑上——有些人半年不开一次。

**错误的做法（Web 思维）**：
```python
# ❌ Web 应用思维：启动时跑 ALTER TABLE
def migrate():
    cursor.execute("ALTER TABLE chats ADD COLUMN model TEXT")  # 如果列已存在就炸
```

**正确的做法（桌面 App 思维）**：
```python
# backend/storage/migrations.py

import sqlite3
from pathlib import Path

class DatabaseMigrator:
    """桌面 App 数据库迁移——使用 PRAGMA user_version 做版本控制"""

    # 迁移脚本注册表：version → 迁移函数
    # ⚠️ 每条迁移必须幂等（重复执行不报错）
    MIGRATIONS = {
        0: "_migrate_v0_to_v1",  # 初始化表结构
        1: "_migrate_v1_to_v2",  # v1.0 → v1.1：新增 user_config 表
        2: "_migrate_v2_to_v3",  # v1.1 → v1.5：chats 表新增 model 列
        # 新增迁移只需在下面加一行，不要改上面的
    }

    def __init__(self, db_path: str):
        self.db = sqlite3.connect(db_path)
        self.db.execute("PRAGMA journal_mode=WAL")  # 迁移期间用 WAL 模式防断电

    def migrate(self):
        current_version = self.db.execute("PRAGMA user_version").fetchone()[0]
        target_version = max(self.MIGRATIONS.keys())

        for version in range(current_version, target_version):
            migration_func = getattr(self, self.MIGRATIONS[version])
            print(f"[迁移] v{version} → v{version + 1}...")
            migration_func()
            self.db.execute(f"PRAGMA user_version = {version + 1}")
            self.db.commit()

        print(f"[迁移] 完成。当前版本: {target_version}")

    # ═══════════════════════════════════════════
    # ⚠️ SQLite 不支持 ALTER COLUMN / DROP COLUMN！
    # 如果要"改列"，只能：
    #   1. 创建新表（含目标结构）
    #   2. INSERT INTO new_table SELECT ... FROM old_table
    #   3. DROP old_table
    #   4. ALTER TABLE new_table RENAME TO old_table
    # ═══════════════════════════════════════════

    def _migrate_v0_to_v1(self):
        """初始表结构"""
        # IF NOT EXISTS 保证幂等
        self.db.executescript("""
            CREATE TABLE IF NOT EXISTS chats (
                id TEXT PRIMARY KEY,
                title TEXT NOT NULL DEFAULT '新对话',
                messages TEXT NOT NULL DEFAULT '[]',  -- JSON 字符串存储（桌面端简单优先）
                model TEXT NOT NULL DEFAULT 'deepseek-chat',
                created_at TEXT NOT NULL DEFAULT (datetime('now')),
                updated_at TEXT NOT NULL DEFAULT (datetime('now'))
            );
            CREATE TABLE IF NOT EXISTS user_config (
                key TEXT PRIMARY KEY,
                value TEXT NOT NULL
            );
            CREATE TABLE IF NOT EXISTS knowledge_files (
                id TEXT PRIMARY KEY,
                file_name TEXT NOT NULL,
                file_path TEXT NOT NULL,
                chunk_count INTEGER NOT NULL DEFAULT 0,
                imported_at TEXT NOT NULL DEFAULT (datetime('now'))
            );
        """)

    def _migrate_v1_to_v2(self):
        """v1.0 → v1.1：新增 user_config 默认值"""
        self.db.executescript("""
            INSERT OR IGNORE INTO user_config (key, value) VALUES ('theme', 'auto');
            INSERT OR IGNORE INTO user_config (key, value) VALUES ('language', 'zh');
            INSERT OR IGNORE INTO user_config (key, value) VALUES ('max_turns', '15');
        """)

    def _migrate_v2_to_v3(self):
        """v1.1 → v1.5：chats 表新增 model 列"""
        # 先检查列是否已存在（幂等）
        existing = self.db.execute("PRAGMA table_info(chats)").fetchall()
        column_names = [col[1] for col in existing]
        if 'model' not in column_names:
            self.db.execute("ALTER TABLE chats ADD COLUMN model TEXT NOT NULL DEFAULT 'deepseek-chat'")
```

**桌面 App 数据库铁律**：
1. **永远不要** `ALTER COLUMN` 或 `DROP COLUMN`——SQLite 不支持。只做 `ADD COLUMN`
2. **每次迁移必须幂等**——用户可能从 v1.0 直接跳到 v1.5
3. **用 `PRAGMA user_version`**——SQLite 内置的版本号，比手动维护 version.txt 可靠
4. **迁移期间用 WAL 模式**——如果用户在迁移中途断电/强杀进程，WAL 可以回滚

#### 4.3.4 Agent 崩溃恢复：工作区事务 + 进度状态管理

> Agent 跑到第 3 轮，用户点了停止（或 Python OOM 闪退）。Rust 重启了 Python，但前端还挂着"思考中"，半成品的临时文件散落在 workspace 里。

**解决方案：Session Workspace + 状态显式管理**

```python
# backend/core/session_manager.py

import uuid
import shutil
from pathlib import Path
from datetime import datetime
from enum import Enum

class AgentStatus(Enum):
    IDLE = "idle"           # 空闲，等待输入
    RUNNING = "running"     # 正在执行 Agent 循环
    FINISHED = "finished"   # 正常完成
    CANCELLED = "cancelled" # 用户手动停止
    ERROR = "error"         # 异常终止（Python 崩溃 / API 超时）

class SessionWorkspace:
    """每次 Agent 任务创建独立的临时工作区"""

    def __init__(self, base_dir: str = "./data/workspaces"):
        self.base_dir = Path(base_dir)
        self.base_dir.mkdir(parents=True, exist_ok=True)
        self.session_id = None
        self.session_dir = None

    def create(self) -> str:
        """开始新任务：创建 session 工作区"""
        self.session_id = f"{datetime.now().strftime('%Y%m%d_%H%M%S')}_{uuid.uuid4().hex[:8]}"
        self.session_dir = self.base_dir / self.session_id
        self.session_dir.mkdir(parents=True, exist_ok=True)
        return self.session_id

    def cleanup(self, completed_successfully: bool = False):
        """任务结束：清理临时文件"""
        if not self.session_dir or not self.session_dir.exists():
            return

        if completed_successfully:
            # 正常完成：把产物移到正式目录，删临时文件
            output_dir = Path("./workspace")
            output_dir.mkdir(exist_ok=True)
            for f in self.session_dir.glob("*"):
                if f.is_file():
                    shutil.move(str(f), str(output_dir / f.name))
        # 不管是否正常完成，都删掉临时工作区
        shutil.rmtree(self.session_dir, ignore_errors=True)

    @staticmethod
    def cleanup_orphaned_sessions(base_dir: str = "./data/workspaces"):
        """启动时清理上次未正常退出的孤儿 session（崩溃残留物）"""
        base = Path(base_dir)
        if not base.exists():
            return
        for session_dir in base.iterdir():
            if session_dir.is_dir():
                shutil.rmtree(session_dir, ignore_errors=True)
```

### 4.4 RAG + 向量数据库 —— 从零落地全流程

> 这是本地知识库功能的技术核心。以下从原理到代码，逐层拆解。

#### 4.4.1 什么是 RAG？为什么需要它？

**RAG（Retrieval-Augmented Generation，检索增强生成）** 解决的核心问题是：

> 大模型的训练数据有截止日期，且不知道你的个人资料。如果你问它"我上周的《数据结构》课件里，二叉树那章讲了什么？"，它无法回答。

RAG 的做法：**先在本地知识库里检索相关内容，再把检索结果一起发给大模型，让它基于这些资料回答。**

```
用户提问
    │
    ▼
┌─────────────────────────────────────────────┐
│  Step 1: 检索 (Retrieval)                   │
│  把问题向量化 → 在向量数据库里找最相似的文档  │
│  → 返回 Top-K 相关片段                       │
└──────────────────┬──────────────────────────┘
                   │
                   ▼  相关文档片段（作为上下文）
┌─────────────────────────────────────────────┐
│  Step 2: 生成 (Generation)                  │
│  System: "根据以下资料回答用户问题..."       │
│  + 检索到的文档片段                          │
│  + 用户原始问题                              │
│  → LLM 生成基于资料的准确回答                │
└─────────────────────────────────────────────┘
```

#### 4.4.2 知识库数据流全景


```python
# backend/knowledge/doc_loader.py

import os
from pathlib import Path

class DocumentLoader:
    """从各种格式中提取纯文本"""

    @staticmethod
    def load(file_path: str) -> str:
        ext = Path(file_path).suffix.lower()
        if ext == '.txt' or ext == '.md':
            return DocumentLoader._load_txt(file_path)
        elif ext == '.pdf':
            return DocumentLoader._load_pdf(file_path)
        elif ext == '.docx':
            return DocumentLoader._load_docx(file_path)
        elif ext == '.pptx':
            return DocumentLoader._load_pptx(file_path)
        else:
            raise ValueError(f"不支持的文件格式: {ext}")

    @staticmethod
    def _load_txt(path: str) -> str:
        with open(path, 'r', encoding='utf-8') as f:
            return f.read()

    @staticmethod
    def _load_pdf(path: str) -> str:
        from pypdf import PdfReader
        reader = PdfReader(path)
        text_parts = []
        for page in reader.pages:
            t = page.extract_text()
            if t:
                text_parts.append(t)
        return '\n\n'.join(text_parts)

    @staticmethod
    def _load_docx(path: str) -> str:
        from docx import Document
        doc = Document(path)
        return '\n\n'.join(p.text for p in doc.paragraphs if p.text.strip())

    @staticmethod
    def _load_pptx(path: str) -> str:
        from pptx import Presentation
        prs = Presentation(path)
        text_parts = []
        for slide in prs.slides:
            for shape in slide.shapes:
                if shape.has_text_frame:
                    text_parts.append(shape.text_frame.text)
        return '\n\n'.join(text_parts)
```

**Step 3：文本分块（Chunking）—— 这是 RAG 质量的关键**

分块策略直接决定检索结果好不好。分太大，检索精度差；分太小，丢失上下文。

```python
# backend/knowledge/chunker.py

class TextChunker:
    """
    多级分块策略：
    - 第一级：按段落（\n\n）自然分界
    - 第二级：按句子（。！？\n）分界
    - 确保每块在 300~800 token 之间
    - 相邻块有 50~100 token 的重叠（overlap）
    """

    def __init__(self, chunk_size: int = 500, overlap: int = 80):
        self.chunk_size = chunk_size      # 目标块大小（字符数）
        self.overlap = overlap            # 重叠量
        # 中文 + 英文的分隔符
        self.separators = [
            "\n\n",     # 段落（最高优先级）
            "\n",       # 换行
            "。",       # 中文句号
            "！",       # 中文感叹号
            "？",       # 中文问号
            ". ",       # 英文句号
            "! ",       # 英文感叹号
            "? ",       # 英文问号
            "；",       # 中文分号
            "; ",       # 英文分号
            ",",        # 逗号
            " ",        # 空格（最低优先级）
        ]

    def chunk(self, text: str) -> list[dict]:
        """返回 [{text, metadata}, ...]"""
        chunks = []
        current_start = 0

        while current_start < len(text):
            # 确定当前块的结束位置
            current_end = current_start + self.chunk_size

            if current_end >= len(text):
                # 最后一块
                chunk_text = text[current_start:]
                chunks.append(self._make_chunk(chunk_text, len(chunks)))
                break

            # 尝试在分隔符处断开（尽量保持语义完整）
            best_break = self._find_best_break(
                text,
                start=current_start + self.chunk_size // 2,  # 最少半块
                end=min(current_end + self.chunk_size // 2, len(text))
            )

            chunk_text = text[current_start:best_break].strip()
            if chunk_text:
                chunks.append(self._make_chunk(chunk_text, len(chunks)))

            # 下一块的起点 = 当前断点 - overlap
            current_start = best_break - self.overlap
            if current_start <= current_start - self.overlap + self.chunk_size:
                # 防止死循环
                current_start = best_break

        return chunks

    def _find_best_break(self, text: str, start: int, end: int) -> int:
        """在 [start, end] 范围内找最佳断开位置"""
        for sep in self.separators:
            # 从后往前找，尽量让块大一些
            pos = text.rfind(sep, start, end)
            if pos != -1:
                return pos + len(sep)
        # 找不到好的分隔符，硬截断
        return end

    def _make_chunk(self, text: str, index: int) -> dict:
        return {
            "text": text,
            "chunk_index": index,
            "char_count": len(text),
        }
```

```python
# backend/knowledge/vector_store.py

import chromadb
from chromadb.config import Settings
from openai import OpenAI
import hashlib
import json

class KnowledgeBase:
    """本地知识库：向量存储 + 语义检索"""

    def __init__(self, persist_dir: str = "./data/chroma_db"):
        self.persist_dir = persist_dir
        # ChromaDB 持久化客户端
        self.client = chromadb.PersistentClient(path=persist_dir)

        # DeepSeek 的 Embedding 接口（兼容 OpenAI 格式）
        self.embed_client = OpenAI(
            api_key="你的DeepSeek-API-Key",    # 实际使用时从配置读取
            base_url="https://api.deepseek.com"
        )
        self.embed_model = "deepseek-chat"     # 或专门的 embedding 模型

    def _get_or_create_collection(self, collection_name: str = "my_knowledge"):
        """获取或创建集合（类似数据库的「表」）"""
        return self.client.get_or_create_collection(
            name=collection_name,
            metadata={"description": "[项目名待定]AI 本地知识库"}
        )

    def _embed(self, texts: list[str]) -> list[list[float]]:
        """将文本转为向量（调用 DeepSeek Embedding API）"""
        resp = self.embed_client.embeddings.create(
            model=self.embed_model,
            input=texts
        )
        return [d.embedding for d in resp.data]

    def add_document(self, file_path: str, file_name: str):
        """导入一个文件到知识库"""
        from .doc_loader import DocumentLoader
        from .chunker import TextChunker

        # 1. 提取文本
        raw_text = DocumentLoader.load(file_path)
        if not raw_text.strip():
            raise ValueError(f"文件内容为空: {file_name}")

        # 2. 分块
        chunker = TextChunker(chunk_size=500, overlap=80)
        chunks = chunker.chunk(raw_text)

        # 3. 向量化
        chunk_texts = [c["text"] for c in chunks]
        embeddings = self._embed(chunk_texts)

        # 4. 存入 ChromaDB
        collection = self._get_or_create_collection()
        doc_hash = hashlib.md5(file_path.encode()).hexdigest()[:8]

        collection.add(
            ids=[f"{doc_hash}_chunk_{c['chunk_index']}" for c in chunks],
            embeddings=embeddings,
            documents=chunk_texts,
            metadatas=[{
                "source_file": file_name,
                "source_path": file_path,
                "chunk_index": c["chunk_index"],
                "char_count": c["char_count"],
                "imported_at": "2026-06-26",  # 实际用 datetime.now()
            } for c in chunks]
        )

        return {"file_name": file_name, "chunks_added": len(chunks)}

    def search(self, query: str, top_k: int = 5) -> list[dict]:
        """语义检索：根据问题找最相关的文档片段"""
        collection = self._get_or_create_collection()

        # 1. 把用户问题向量化
        query_embedding = self._embed([query])[0]

        # 2. 在 ChromaDB 中检索
        results = collection.query(
            query_embeddings=[query_embedding],
            n_results=top_k,
            include=["documents", "metadatas", "distances"]
        )

        # 3. 整理返回结果
        hits = []
        for i in range(len(results["ids"][0])):
            hits.append({
                "text": results["documents"][0][i],
                "source_file": results["metadatas"][0][i]["source_file"],
                "relevance_score": round(1 - results["distances"][0][i], 4),
            })
        return hits
```

**Step 5：RAG 问答 —— 检索 + 生成**

```python
# backend/knowledge/rag_qa.py

class RAGQuestionAnswerer:
    """基于知识库的 RAG 问答"""

    SYSTEM_PROMPT_TEMPLATE = """你是[项目名待定]AI，一个大学生的学习助手。

## 回答规则
1. 只根据下面「参考资料」中的内容回答问题
2. 如果资料中找不到答案，诚实告诉用户"我不确定，资料中没提到这个"
3. 回答时注明引用的来源文件名
4. 如果资料中有多个相关段落，归纳总结，不要直接复制粘贴
5. 用清晰的、大学生能看懂的语言回答

## 参考资料
{context}

## 用户问题
{question}

请回答："""

    def __init__(self, knowledge_base: KnowledgeBase, chat_client):
        self.kb = knowledge_base
        self.chat_client = chat_client  # 大模型客户端（DeepSeek/Claude/等）

    async def ask(self, question: str, top_k: int = 5) -> dict:
        # Step 1: 检索相关文档片段
        hits = self.kb.search(question, top_k=top_k)

        if not hits:
            return {
                "answer": "你的知识库里还没有相关内容。先导入一些课件或资料吧！",
                "sources": []
            }

        # Step 2: 构造上下文
        context_parts = []
        for i, hit in enumerate(hits):
            context_parts.append(
                f"[资料{i+1}] 来源: {hit['source_file']}\n{hit['text']}"
            )
        context = "\n\n---\n\n".join(context_parts)

        # Step 3: 构造完整 Prompt
        prompt = self.SYSTEM_PROMPT_TEMPLATE.format(
            context=context,
            question=question
        )

        # Step 4: 调用 LLM 生成答案（流式返回）
        response = await self.chat_client.chat(
            messages=[{"role": "user", "content": prompt}],
            stream=True
        )

        return {
            "answer": response,           # 可流式
            "sources": list(set(h["source_file"] for h in hits))  # 去重
        }
```

```python
async def upload_file(file: UploadFile = File(...)):
    """导入文件到知识库"""
    import tempfile, os
    # 保存上传的文件到临时目录
    with tempfile.NamedTemporaryFile(delete=False, suffix=os.path.splitext(file.filename)[1]) as tmp:

    rag = RAGQuestionAnswerer(kb, chat_client)

    async def generate():
        result = await rag.ask(req.question, req.top_k)
        # 先发送来源信息

        # 再流式发送答案
        async for chunk in result["answer"]:
            yield f"data: {json.dumps({'type': 'text', 'content': chunk})}\n\n"
        yield "data: [DONE]\n\n"

    return StreamingResponse(generate(), media_type="text/event-stream")

    """列出已导入的文件"""
    collection = kb._get_or_create_collection()
    # 从 metadata 中提取唯一文件名
    all_meta = collection.get(include=["metadatas"])
    files = list(set(
        m["source_file"] for m in all_meta["metadatas"] if m
    ))
    return {"files": files}

@router.delete("/files/{file_name}")
async def delete_file(file_name: str):
    """从知识库中删除某个文件的所有分块"""
    collection = kb._get_or_create_collection()
    # 查询该文件的所有 chunk id
    all_data = collection.get(include=["metadatas"])
    ids_to_delete = [
        id for id, meta in zip(all_data["ids"], all_data["metadatas"])
        if meta and meta.get("source_file") == file_name
    ]
    if ids_to_delete:
        collection.delete(ids=ids_to_delete)
    return {"status": "deleted", "chunks_removed": len(ids_to_delete)}
```

#### 4.4.5 关键技术细节与避坑指南

**1. 分块策略直接影响 RAG 质量**：
- 课件类文档（PPT→文字）：每页 PPT 一般是一个独立的知识点，建议**按页分块**
- 教材类文档（PDF/Word）：建议按段落 + 固定窗口混合分块，overlap 不要太小（建议 80~120 字符）
- 对话记录/笔记：时间线结构，建议按时间戳分块

**2. Embedding 成本控制**：
- 一份 100 页的 PDF 大约 20 万中文字符 → 分成约 400~500 个块 → 一次 embedding 大约 10 万 token
- DeepSeek Embedding ¥0.25/百万 token → **导入一本教材的成本约 ¥0.025**，几乎可以忽略不计
- 注意做去重和缓存：同一文件不要重复 embedding

**3. 混合检索（Hybrid Search）**：
> 纯向量检索有时会漏掉关键字的精确匹配。比如用户问"第三章的课后习题"，向量检索可能返回语义相似的第四章内容，但用户要的是精确的"第三章"。

解决办法（P2 阶段加入）：
```
语义相似度得分 (0.7) + 关键词匹配得分 (0.3) = 最终排序分
```

实战中用 ChromaDB 的 `where` 条件做 metadata 过滤 + embedding 语义检索，就是对混合检索的简化实现。

**4. 检索后重排序（Re-ranking）**：
> Top-K 拿回来的文档片段中，可能有表面相似但实际无关的内容。在发给 LLM 之前，用 Cross-Encoder 模型再筛一次。

P2 阶段建议引入，简单的重排模型如 `BAAI/bge-reranker-v2-m3`（开源，免费，中文效果好）。

**5. 引用溯源**：
用户最关心"AI 说的对不对"。每个答案必须附上来源文件名，最好能定位到具体段落。上面的 Prompt 模板已经加了来源标注，前端展示时做成可点击的链接，让用户能打开原始文件验证。

#### 4.6.1 先搞清楚：Office 文件本质是什么？

```

  "你好世界" → 存的就是这 4 个字的 UTF-8 字节 → 直接读

PDF 文件：
  页面描述语言 → 一堆"在(x,y)画字符'你'，字体SimSun，大小12pt"的指令
  → 需要解析引擎才能还原出文字

DOCX / PPTX / XLSX 文件：
  ZIP 压缩包 → 里面是一堆 XML 文件 → 需要理解 Office 的 XML Schema
  → 解压→改 XML→重新打包 ZIP 就能操作
```

**一句话**：PDF 是渲染指令集，Office 新格式本质是 ZIP 里的 XML。理解了这个，下面的所有操作都顺了。

#### 4.6.2 PDF 读取：把"打印指令"还原成文字

##### 原理
PDF 不是一个"文本容器"，而是一个"电子纸"。它记录的是"在页面的什么位置画什么字符"。文字提取需要**逆向工程**这个过程。

##### 两条技术路线

| 路线 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **pypdf (PyPDF2)** | 直接解析 PDF 内部的文本流对象 | 快、纯 Python、无依赖 | 双栏 PDF、扫描件、特殊字体提取效果差 |
| **pdfplumber** | 基于 PDF 的坐标信息重建文本布局 | 表格提取极强、中文支持好 | 慢一些、内存占用大 |

##### 实战代码（处理你同学会遇到的各种情况）

```python
# backend/knowledge/pdf_reader.py

import pdfplumber
from pypdf import PdfReader
import os

class PDFReader:
    """PDF 读取器：自动选择最优策略"""

    @staticmethod
    def extract_text(file_path: str, strategy: str = "auto") -> str:
        """
        strategy:
          - "auto": 先用 pypdf 快速提取，如果结果太短则回退到 pdfplumber
          - "text": 纯文本层提取（pypdf，快）
          - "layout": 布局感知提取（pdfplumber，准）
        """
        if strategy == "text":
            return PDFReader._extract_with_pypdf(file_path)
        elif strategy == "layout":
            return PDFReader._extract_with_pdfplumber(file_path)
        else:  # auto
            text = PDFReader._extract_with_pypdf(file_path)
            # 如果 pypdf 出来的字数 < 文件大小的 1/500，说明可能有问题
            file_size = os.path.getsize(file_path)
            if len(text) < file_size / 500:
                return PDFReader._extract_with_pdfplumber(file_path)
            return text

    @staticmethod
    def _extract_with_pypdf(path: str) -> str:
        reader = PdfReader(path)
        parts = []
        for i, page in enumerate(reader.pages):
            text = page.extract_text()
            if text:
                parts.append(f"[第{i+1}页]\n{text}")
        return '\n\n'.join(parts)

    @staticmethod
    def _extract_with_pdfplumber(path: str) -> str:
        parts = []
        with pdfplumber.open(path) as pdf:
            for i, page in enumerate(pdf.pages):
                text = page.extract_text()
                if text:
                    parts.append(f"[第{i+1}页]\n{text}")

                # 额外：提取表格
                tables = page.extract_tables()
                for j, table in enumerate(tables):
                    parts.append(f"[第{i+1}页 表格{j+1}]")
                    for row in table:
                        parts.append(' | '.join(str(c) if c else '' for c in row))
        return '\n\n'.join(parts)
```

##### 踩坑清单

| 坑 | 现象 | 解决 |
|----|------|------|
| **扫描版 PDF** | 提取出来全是空 | 本质是图片，需要 OCR（用 PaddleOCR 或 Tesseract 预处理）。这个先不强求，提示用户"扫描件不支持"就行 |
| **双栏论文** | 文字顺序全乱，左栏一行右栏一行混在一起 | pdfplumber 对双栏支持比较好，但也不能100%解决。学术论文场景建议提示用户"结果可能需要对读" |
| **数学公式** | 公式变成乱码或丢失 | 无解。公式在 PDF 里是内嵌字体/图片。需要 OCR 数学公式专用模型（如 MathPix API），成本太高，P3 再考虑 |
| **加密 PDF** | 直接报错 | pypdf 有 `decrypt()` 方法，但只支持简单密码。强加密无解 |
| **超大文件** | 内存爆了 | 逐页读取 + 即时写入 ChromaDB，不要一次性把整本书读到内存 |

#### 4.6.3 PPT 读取：从幻灯片里抠文字

##### 原理
PPTX 就是 ZIP。结构如下：
```
演示文稿.pptx
├── [Content_Types].xml
├── ppt/
│   ├── slides/
│   │   ├── slide1.xml    ← 每一页幻灯片
│   │   ├── slide2.xml
│   │   └── ...
│   ├── notesSlides/
│   │   ├── notesSlide1.xml  ← 讲师备注（有时候重要内容在这里！）
│   │   └── ...
│   └── media/            ← 图片/视频（我们暂时不管）
└── docProps/
```

python-pptx 库帮我们封装了这些 XML 解析，但你需要知道它**不只是提取文字这么简单**——PPT 里的文字分布在各种形状（Shape）里：文本框、表格、图表、SmartArt……

##### 实战代码

```python
# backend/knowledge/ppt_reader.py

from pptx import Presentation
from pptx.util import Inches

class PPTReader:
    """PPT 读取器：提取所有可读文本，保留结构信息"""

    @staticmethod
    def extract_text(file_path: str) -> str:
        prs = Presentation(file_path)
        parts = []
        slide_num = 0

        for slide in prs.slides:
            slide_num += 1
            slide_parts = [f"\n=== 第{slide_num}页 ==="]

            for shape in slide.shapes:
                # 文本框
                if shape.has_text_frame:
                    text = shape.text_frame.text.strip()
                    if text:
                        slide_parts.append(text)

                # 表格
                if shape.has_table:
                    table = shape.table
                    rows = []
                    for row in table.rows:
                        cells = [cell.text.strip() for cell in row.cells]
                        rows.append(' | '.join(cells))
                    slide_parts.append('\n'.join(rows))

                # 图表（只能提取标题和数据标签文字）
                if shape.has_chart:
                    chart = shape.chart
                    if chart.has_title:
                        slide_parts.append(f"[图表] {chart.chart_title.text_frame.text}")

            # 讲师备注（老师课件里常常有重要补充）
            if slide.has_notes_slide:
                notes = slide.notes_slide.notes_text_frame.text.strip()
                if notes:
                    slide_parts.append(f"[讲师备注] {notes}")

            parts.append('\n'.join(slide_parts))

        return '\n\n'.join(parts)

    @staticmethod
    def extract_by_slides(file_path: str) -> list[dict]:
        """按幻灯片分页提取（用于精确检索——用户问某页内容时能定位到具体页）"""
        prs = Presentation(file_path)
        slides_data = []

        for i, slide in enumerate(prs.slides):
            texts = []
            for shape in slide.shapes:
                if shape.has_text_frame:
                    t = shape.text_frame.text.strip()
                    if t:
                        texts.append(t)

            slides_data.append({
                "slide_number": i + 1,
                "text": '\n'.join(texts),
                "char_count": sum(len(t) for t in texts),
            })

        return slides_data
```

##### 踩坑清单

| 坑 | 现象 | 解决 |
|----|------|------|
| **.ppt 老格式** | 直接报错 | `.ppt`（97-2003格式）是二进制，不是 ZIP。用 LibreOffice 命令行转成 `.pptx` 再读。在代码里做格式判断：遇到 `.ppt` 就提示用户转换 |
| **文字在图片里** | 提取出来是空 | 无解。很多老师的课件就是"图片+几个标题"，实际内容全在图上。这种情况告诉用户"该 PPT 以图片为主，暂不支持提取" |
| **SmartArt 图形** | 文字丢失 | python-pptx 对 SmartArt 支持有限。SmartArt 里的文字在 XML 的 `dgm:shape` 节点下，需要用 `lxml` 直接解析 XML |
| **内嵌视频/音频** | 跳过 | 暂时不管，多媒体内容不影响文本检索 |

#### 4.6.4 Word 文档：读和写都要做好

##### 原理
DOCX 也是 ZIP：
```
论文.docx
├── word/
│   ├── document.xml   ← 正文内容
│   ├── styles.xml     ← 样式定义（标题/正文/引用）
│   ├── comments.xml   ← 批注
│   └── media/         ← 嵌入图片
```

##### Word 读取

```python
# backend/knowledge/docx_reader.py

from docx import Document

class DocxReader:
    @staticmethod
    def extract_text(file_path: str) -> str:
        doc = Document(file_path)
        parts = []

        for para in doc.paragraphs:
            # 获取段落样式（区分标题/正文）
            style = para.style.name if para.style else "Normal"
            text = para.text.strip()
            if not text:
                continue

            if 'Heading' in style or 'Title' in style or '标题' in style:
                parts.append(f"## {text}")
            else:
                parts.append(text)

        # 也提取表格
        for table in doc.tables:
            parts.append('\n---\n')
            for row in table.rows:
                cells = [cell.text.strip() for cell in row.cells]
                parts.append(' | '.join(cells))

        return '\n\n'.join(parts)
```

##### Word 生成——这是重点

生成 Word 不是"把文本塞进 docx 就完了"。同学要的是**格式正确的论文/报告**，有封面、目录、标题层级、参考文献格式。

```python
# backend/tools/word_generator.py

from docx import Document
from docx.shared import Pt, Inches, Cm, RGBColor
from docx.enum.text import WD_ALIGN_PARAGRAPH
from docx.oxml.ns import qn
import datetime

class WordGenerator:
    """
    论文/报告生成器

    核心思路：
    AI 生成内容（文本） + 你定义模板（格式） = 格式正确的文档
    不要指望 AI 直接输出格式完美的 Word——它做不到
    """

    @staticmethod
    def create_essay(
        title: str,
        author: str,
        sections: list[dict],  # [{"heading": "摘要", "body": "本文..."}, ...]
        output_path: str
    ):
        """
        sections 结构：
        [
            {"heading": "摘要", "body": "本文研究了..."},
            {"heading": "一、引言", "body": "在当今..."},
            {"heading": "二、文献综述", "body": "国内外学者..."},
            ...
        ]
        """
        doc = Document()

        # ── 页面设置 ──
        section = doc.sections[0]
        section.page_width = Cm(21.0)    # A4
        section.page_height = Cm(29.7)
        section.top_margin = Cm(2.54)
        section.bottom_margin = Cm(2.54)

        # ── 封面（手动拼） ──
        # 空几行做居中效果
        for _ in range(6):
            doc.add_paragraph('')

        title_para = doc.add_paragraph()
        title_para.alignment = WD_ALIGN_PARAGRAPH.CENTER
        run = title_para.add_run(title)
        run.font.size = Pt(22)
        run.font.bold = True
        run.font.name = '黑体'
        run._element.rPr.rFonts.set(qn('w:eastAsia'), '黑体')

        doc.add_paragraph('')

        author_para = doc.add_paragraph()
        author_para.alignment = WD_ALIGN_PARAGRAPH.CENTER
        run = author_para.add_run(author)
        run.font.size = Pt(14)
        run.font.name = '宋体'
        run._element.rPr.rFonts.set(qn('w:eastAsia'), '宋体')

        doc.add_page_break()

        # ── 正文各章节 ──
        for sec in sections:
            heading = sec.get('heading', '')
            body = sec.get('body', '')

            # 标题
            h = doc.add_heading(heading, level=1)
            for run in h.runs:
                run.font.name = '黑体'
                run._element.rPr.rFonts.set(qn('w:eastAsia'), '黑体')

            # 正文段落
            paragraphs = body.split('\n')
            for p_text in paragraphs:
                if not p_text.strip():
                    continue
                p = doc.add_paragraph()
                p.paragraph_format.first_line_indent = Cm(0.74)  # 首行缩进2字符
                p.paragraph_format.line_spacing = 1.5
                run = p.add_run(p_text.strip())
                run.font.size = Pt(12)
                run.font.name = '宋体'
                run._element.rPr.rFonts.set(qn('w:eastAsia'), '宋体')

        # ── 参考文献 ──
        doc.add_heading('参考文献', level=1)
        return doc

        doc.save(output_path)
        return output_path
```

##### Word 生成的踩坑清单

| 坑 | 现象 | 解决 |
|----|------|------|
| **中文字体不生效** | 代码设置了"宋体"但打开是默认字体 | 必须同时设置 `font.name`（西文）和 `w:eastAsia`（东亚字体），如代码所示 |
| **目录生成** | 手动生成目录极繁琐 | 用 `python-docx` 做不到自动目录（需要 Word 内部渲染）。替代方案：先生成文档，然后提示用户"在 Word 里右键更新目录" |
| **页眉页脚** | 默认没有 | 用 `section.header` 和 `section.footer` 设置，但页码需要添加 `w:fldChar` 的 XML 元素（有点复杂，建议 MVP 阶段跳过） |
| **参考文献格式** | GB/T 7714 格式复杂 | 让 AI 生成参考文献文本即可，格式不要追求完美——老师也不会逐字查格式 |

#### 4.6.5 PPT 生成：不只是"往幻灯片里填字"

##### 这个功能比你想象的复杂

PPT 不是纯文本——它涉及**布局、排版、视觉层次**。你不能像生 Word 那样"AI 写一段话，塞进一页 PPT"。你需要：
1. AI 生成**结构**（标题 + 要点 + 配图建议）
2. 代码负责**渲染**（选布局模板 → 填文字 → 调字号 → 配色）

##### 推荐落地路径

```
MVP 阶段（够用就行）:
  AI 生成大纲 → 套用固定模板 → 填文字 → 导出

P2 阶段（好看一点）:
  用户选风格 → AI 生成每页内容结构 → 多种布局模板 → 自动排版
```

##### 简洁版生成代码

```python
# backend/tools/ppt_generator.py

from pptx import Presentation
from pptx.util import Inches, Pt, Cm
from pptx.dml.color import RGBColor
from pptx.enum.text import PP_ALIGN

class PPTGenerator:
    """PPT 生成器 —— MVP 版用固定模板"""

    # 色板（匹配你的 UI 设计）
    PRIMARY = RGBColor(0x50, 0x58, 0xA8)    # #5058A8
    ACCENT = RGBColor(0x8C, 0x2E, 0x50)     # #8C2E50
    TEXT_DARK = RGBColor(0x1A, 0x16, 0x26)  # #1A1626
    BG_LIGHT = RGBColor(0xF8, 0xFA, 0xFF)   # #F8FAFF

    @staticmethod
    def create_presentation(
        title: str,
        slides: list[dict],  # [{"title": "...", "bullets": ["...", "..."], "note": "..."}, ...]
        output_path: str
    ):
        """
        slides 结构（AI 返回的格式）:
        [
            {
                "title": "二叉树定义",
                "bullets": ["每个节点最多两个子节点", "左子树和右子树有序", "递归定义的数据结构"],
                "note": "这里强调和'树'的区别——二叉是有序的"
            },
            ...
        ]
        """
        prs = Presentation()
        prs.slide_width = Inches(13.333)   # 16:9 宽屏
        prs.slide_height = Inches(7.5)

        # ── 封面 ──
        slide = prs.slides.add_slide(prs.slide_layouts[6])  # 空白布局
        # 背景色
        bg = slide.background
        bg.fill.solid()
        bg.fill.fore_color.rgb = PPTGenerator.BG_LIGHT

        # 标题
        txBox = slide.shapes.add_textbox(Inches(1.5), Inches(2.5), Inches(10), Inches(1.5))
        tf = txBox.text_frame
        p = tf.paragraphs[0]
        p.text = title
        p.font.size = Pt(36)
        p.font.bold = True
        p.font.color.rgb = PPTGenerator.PRIMARY
        p.alignment = PP_ALIGN.CENTER

        # ── 内容页 ──
        for slide_data in slides:
            slide = prs.slides.add_slide(prs.slide_layouts[6])

            # 页标题
            txBox = slide.shapes.add_textbox(Inches(0.8), Inches(0.5), Inches(11.5), Inches(0.8))
            tf = txBox.text_frame
            p = tf.paragraphs[0]
            p.text = slide_data["title"]
            p.font.size = Pt(28)
            p.font.bold = True
            p.font.color.rgb = PPTGenerator.PRIMARY

            # 分隔线
            line = slide.shapes.add_shape(
                1,  # MSO_SHAPE.RECTANGLE
                Inches(0.8), Inches(1.4), Inches(2), Inches(0.04)
            )
            line.fill.solid()
            line.fill.fore_color.rgb = PPTGenerator.ACCENT
            line.line.fill.background()  # 无边框

            # 要点
            txBox = slide.shapes.add_textbox(Inches(0.8), Inches(1.8), Inches(11), Inches(4.5))
            tf = txBox.text_frame
            tf.word_wrap = True

            for i, bullet in enumerate(slide_data.get("bullets", [])):
                if i == 0:
                    p = tf.paragraphs[0]
                else:
                    p = tf.add_paragraph()
                p.text = f"• {bullet}"
                p.font.size = Pt(18)
                p.font.color.rgb = PPTGenerator.TEXT_DARK
                p.space_after = Pt(12)

            # 讲师备注
            if slide_data.get("note"):
                slide.notes_slide.notes_text_frame.text = slide_data["note"]

        prs.save(output_path)
        return output_path
```

##### PPT 生成踩坑清单

| 坑 | 深度 | 建议 |
|----|------|------|
| **排版"丑"** | 自动生成的 PPT 肯定比人手做的丑 | 承认这个现实。策略是"内容对 + 格式整洁 > 花哨但内容空"。给用户 3 种模板风格可选，不要指望 AI 自动做出精美的幻灯片 |
| **图片/图表插入** | 位置、大小、比例都需要精确控制 | MVP 阶段不做，只做纯文字 PPT。P2 阶段支持 AI 生成配图提示词 → 调生图 API → 插入 PPT |
| **SmartArt / 图示** | python-pptx 不直接支持 SmartArt 创建 | 用 `python-pptx` 的 `shapes.add_shape()` 手动拼矩形/箭头/圆，做简单的流程图 |

#### 4.6.6 Excel 生成：不只是导出 CSV

同学需要的 Excel：
- 课程表 → 带颜色的单元格合并
- 数据统计 → 带公式的表格
- 成绩分析 → 带图表的 spreadsheet

```python
# backend/tools/excel_generator.py

import openpyxl
from openpyxl.styles import Font, PatternFill, Alignment, Border, Side
from openpyxl.utils import get_column_letter

class ExcelGenerator:
    """表格生成器"""

    # 样式库（复用你的 UI 色彩）
    HEADER_FILL = PatternFill(start_color="5058A8", end_color="5058A8", fill_type="solid")
    HEADER_FONT = Font(name="微软雅黑", size=11, bold=True, color="FFFFFF")
    ACCENT_FILL = PatternFill(start_color="F8FAFF", end_color="F8FAFF", fill_type="solid")
    BODY_FONT = Font(name="微软雅黑", size=10)

    @staticmethod
    def create_table(
        headers: list[str],
        rows: list[list],
        title: str = "",
        output_path: str = "output.xlsx"
    ):
        wb = openpyxl.Workbook()
        ws = wb.active
        ws.title = title or "Sheet1"

        # 标题行
        if title:
            ws.merge_cells(start_row=1, start_column=1, end_row=1, end_column=len(headers))
            ws.cell(1, 1, title).font = Font(name="微软雅黑", size=14, bold=True,
                                              color="5058A8")
            ws.cell(1, 1).alignment = Alignment(horizontal="center")
            start_row = 3
        else:
            start_row = 1

        # 表头
        for col, header in enumerate(headers, 1):
            cell = ws.cell(start_row, col, header)
            cell.fill = ExcelGenerator.HEADER_FILL
            cell.font = ExcelGenerator.HEADER_FONT
            cell.alignment = Alignment(horizontal="center")

        # 数据行
        for r, row in enumerate(rows):
            for c, value in enumerate(row):
                cell = ws.cell(start_row + 1 + r, c + 1, value)
                cell.font = ExcelGenerator.BODY_FONT
                if r % 2 == 0:
                    cell.fill = ExcelGenerator.ACCENT_FILL  # 斑马纹

        # 自动列宽
        for col in range(1, len(headers) + 1):
            ws.column_dimensions[get_column_letter(col)].width = 15

        wb.save(output_path)
        return output_path
```

#### 4.6.7 Office 文件操作的架构总结

```
用户操作
    │
    ├── "帮我读这个课件" ──→ PDFReader / PPTReader / DocxReader
    │                        └──→ 文本提取
    │                             └──→ 分块 → Embedding → ChromaDB
    │                                  └──→ 后续可用 RAG 问答
    │
    ├── "帮我写一篇论文" ──→ LLM 生成内容
    │                        └──→ WordGenerator.create_essay()
    │                             └──→ .docx 文件 → 自动打开
    │
    ├── "帮我做个PPT"  ──→ LLM 生成大纲和每页要点
    │                        └──→ PPTGenerator.create_presentation()
    │                             └──→ .pptx 文件 → 自动打开
    │
    └── "帮我做个表格" ──→ LLM 生成结构化数据
                             └──→ ExcelGenerator.create_table()
                                  └──→ .xlsx 文件 → 自动打开
```

### 4.7 "说简单做难"：功能背后的隐藏复杂度

> 以下每个功能，说出来只要 5 秒，但做起来可能花 5 天甚至 5 周。提前知道坑在哪，比掉进去再爬出来强。

#### 4.7.1 "AI 帮我写论文"——难的不是写，是控制

| 你以为的 | 实际的 | 为什么难 |
|---------|--------|---------|
| AI 一次性写出 5000 字 | AI 一次只能稳定输出 ~800-1500 字，超出容易跑偏 | Token 限制 + 模型注意力衰减。需要**分章节生成**，每章带上一章的摘要做上下文 |
| 引用真实参考文献 | AI 编造不存在的论文 | LLM 的"幻觉"问题。真实引用需要**对接学术搜索 API**（知网/Google Scholar），或者用 RAG 从用户导入的文献里找 |
| 格式符合学校模板 | 生成出来的排版乱七八糟 | LLM 不理解 Word 排版。必须用**代码控制格式**（见 4.6.4），LLM 只负责文字内容 |
| 查重率低 | 大量模板化表达 | 本科生论文尤其容易。需要**多次改写**策略：生成 → 查重 → 标出重复段 → 改写 → 再查 |
| 中文标点、语法正确 | 偶尔出现英文标点或语法错误 | 模型主要训练数据是英文。需要在 System Prompt 里强调标点规则，外加后处理正则修正 |

**落地策略**：不要做成"一键生成论文"，而是"分步引导"——

```
Step 1: 确定题目和关键词 → AI 生成大纲 → 用户审核修改
Step 2: 逐章节展开 → 每次只生成 1 节（500~800 字） → 用户修改确认
Step 3: 生成摘要和结论 → 汇总整合
Step 4: 自动排版 → 用户最终检查
```

#### 4.7.2 "AI 整理错题"——难的是结构化

| 你以为的 | 实际的 | 为什么难 |
|---------|--------|---------|
| 拍个照 → AI 解析 → 自动整理 | 照片质量参差不齐：斜的、模糊的、手写潦草的 | 需要先用 Vision API 识别，但**手写中文的 OCR 准确率**远低于印刷体 |
| 自动按科目/知识点分类 | 一道题可能跨多个知识点，边界模糊 | 需要维护一个**知识点标签体系**，且要支持多标签。分类调 Prompt 是个迭代活 |
| 定期提醒复习 | 间隔重复算法（艾宾浩斯）的实现不简单 | SM-2 算法本身不难（几十行代码），难的是**什么算"掌握"、什么算"忘记"的判定逻辑** |

**落地策略**：
- 第一版只支持**文字输入题目**（用户手动打/粘贴），跳过 OCR 这个坑
- 第二版加上**截图识别**（印刷体题目），用 Vision API
- 第三版再搞**手写识别**（需要专用 OCR）

#### 4.7.3 "AI 读课件然后回答问题"——难的是检索精度

这个在 4.4 节已经详细拆解了。额外补充一点：

> RAG 的质量 = 分块策略 × 检索策略 × Prompt 设计。三个乘数任何一个为 0，结果就是 0。这也是为什么很多产品"看起来做了 RAG，但是完全不好用"的原因。

#### 4.7.4 "全局快捷键唤起"——难的是系统兼容性

| 问题 | 细节 |
|------|------|
| 不同 Windows 版本行为不同 | Win10 和 Win11 的快捷键注册 API 行为有细微差异 |
| 杀毒软件误报 | 全局键盘钩子经常被 360/腾讯管家误认为是"键盘记录木马" |
| 与其他软件冲突 | 如果用户已经用 Alt+Space（比如 PowerToys Run 或 Everything），你的快捷键就注册不上 |
| 以管理员权限运行时快捷键失效 | Windows UAC 机制：你的普通权限窗口 vs 管理员权限的别的窗口，有时按键事件穿透不了 |

**落地策略**：
- C++ 用 `RegisterHotKey` API（最兼容的方式）
- 支持用户自定义快捷键（在设置里改），避免冲突
- 第一次启动时检测快捷键是否注册成功，失败了给明确提示

#### 4.7.5 "流式打字机效果"——你用 SSE 就对了，但有两个细节

```javascript
// 大多数人这么写——有 bug
const response = await fetch('/api/chat', { body: ... });
const reader = response.body.getReader();
while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    const text = new TextDecoder().decode(value);
    chatBox.innerHTML += text;  // ← BUG：每次 += 整个 HTML 都会被重新解析
}

// 应该这么做
let buffer = '';
const MARKDOWN_BREAKS = ['\n\n', '。', '！', '？', '. ', '! ', '? '];
while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += new TextDecoder().decode(value);

    // 在合适的断点才触发 Markdown 渲染，而不是每来一个字就渲染一次
    if (MARKDOWN_BREAKS.some(sep => buffer.endsWith(sep)) || buffer.length > 200) {
        renderMarkdown(chatBox, buffer);  // 累积够一个整句再渲染
    }
}
```

原因：Markdown 渲染（marked.js）的 CPU 开销不低。每来一个字就全量重渲染，响应的字符数一多就会卡顿。用**缓冲 + 断句渲染**，前端体验丝滑 10 倍。

#### 4.7.6 "多模型切换"——难在 Prompt 的跨模型适配

同一个 Prompt 发给 DeepSeek、Claude、GPT，结果可能天差地别：

| 模型 | 风格特点 | 需要注意 |
|------|---------|---------|
| DeepSeek | 中文极好，逻辑性强，但有时过于"理工直男" | System Prompt 里加"请用温和亲切的语气"会更适合学生场景 |
| Claude | 长文理解最强，但有时过于谨慎（拒绝回答） | 避免使用"破解"、"绕过"等触发词 |
| GPT | 创意性强，但中文偶尔不如国产模型自然 | 设定 language preference |

**落地策略**：为每个模型维护一套**精简版 System Prompt**（不是复制粘贴三份，而是核心指令 + 模型特定的微调指令）。这个在 `model_router.py` 里做，不复杂但容易被忽略。

#### 4.7.7 "Python 打包成 exe"——PyInstaller 是魔鬼

PyInstaller 的典型痛苦路径：
```
第1次: 打包 → 800MB → 太大了！
第2次: 加 --exclude → 500MB → 能接受
第3次: 发给朋友 → 闪退 → 没有日志不知道原因
第4次: 加 --debug → 发现是 chromadb 的 C 扩展没打进去
第5次: 手动指定 hidden-imports → 终于能跑了
第6次: 朋友说报"XXX.dll 缺失" → ...
```

建议的打包策略：
- Python 部分就用 **PyInstaller** 打包成一个文件夹（不是单 exe，调试方便）
- Rust/Tauri 壳部分用 `cargo tauri build` 打包
- 最终目录结构：


### 4.8 Agent 核心循环：从"调 API"到"自主干活"

> 你以前做的：用户输入 → 调 AI → AI 返回 JSON → 代码按 JSON 执行 → 结束。
> Agent 要做的：用户输入 → AI 思考 → 调工具 → 看结果 → 再思考 → 再调工具 → ... → 直到完成任务。
>
> 区别在于**循环**。一个回合不够就再来一个回合，AI 自己决定什么时候停。

#### 4.8.1 最简 Agent 循环（12 行伪代码）

```python
async def agent_loop(user_input: str, tools: list[dict], max_turns: int = 20):
    messages = [{"role": "system", "content": SYSTEM_PROMPT},
                {"role": "user", "content": user_input}]

    for turn in range(max_turns):
        response = await llm.chat(messages, tools=tools)

        if response.has_tool_calls():
            for tool_call in response.tool_calls:
                result = await execute_tool(tool_call)      # 执行工具
                messages.append(tool_call.to_message())      # "我调用了 xxx"
                messages.append({"role": "tool", "content": result})  # "结果是 yyy"
            continue  # ← 回到循环开头，让 AI 看结果后继续思考

        # 没有工具调用 = AI 觉得任务完成了
        return response.content
```

**整段逻辑就是**：只要 AI 还想调工具，就让它调，调完把结果喂回去，让它决定下一步。直到 AI 觉得不需要再调工具了，返回最终答案。

##### 组件 3：Agent 调度器 —— 把一切串起来

```python
# backend/core/agent_dispatcher.py

import asyncio
import json
import time
import traceback
from typing import AsyncGenerator

class AgentDispatcher:
    """
    Agent 核心调度器。
    实现完整的 Think → Act → Observe → Think 循环。
    """

    def __init__(self, llm_client, tool_executor, max_turns: int = 15):
        self.llm = llm_client
        self.tools = tool_executor
        self.max_turns = max_turns
        self.system_prompt = AGENT_SYSTEM_PROMPT

    async def run(
        self,
        user_input: str,
        history: list[dict] = None,
        model: str = "deepseek-chat"
    ) -> AsyncGenerator[str, None]:
        """
        流式执行 Agent 循环。
        每个 yield 都是一条 SSE 事件，前端实时展示。
        """
        # 初始化消息列表
        messages = [{"role": "system", "content": self.system_prompt}]

        # 加入历史对话（最近 N 轮）
        if history:
            messages.extend(self._prepare_history(history))

        # 加入当前用户输入
        messages.append({"role": "user", "content": user_input})

        turn = 0
        total_tool_calls = 0

        while turn < self.max_turns:
            turn += 1

            # ── Token 预算检查 ──
            token_count = self._estimate_tokens(messages)
            if token_count > 50000:  # 超过 50K token 就压缩
                yield self._sse("status", f"上下文过长，正在自动压缩...")
                messages = self._compress_context(messages)
                yield self._sse("status", f"压缩完成，当前 token 数约 {self._estimate_tokens(messages)}")

            # ── 调用 LLM ──
            yield self._sse("status", f"思考中... (第{turn}轮)")

            try:
                response = await self.llm.chat(
                    messages=messages,
                    tools=ToolDefinition.get_tools_for_model(model),
                    model=model,
                    stream=False  # Agent 循环阶段不用流式，先拿到完整结果
                )
            except Exception as e:
                yield self._sse("error", f"模型调用失败: {str(e)}")
                return

            # ── 判断返回类型 ──
            if response.tool_calls:
                # AI 想调工具
                for tc in response.tool_calls:
                    total_tool_calls += 1
                    yield self._sse("tool_start", f"正在执行: {tc.function.name}")

                    # 安全检查
                    if not self._safety_check(tc):
                        yield self._sse("tool_error", f"安全策略阻止了操作: {tc.function.name}")
                        messages.append({
                            "role": "tool",
                            "tool_call_id": tc.id,
                            "content": "[拒绝] 此操作被安全策略阻止。请尝试其他方法。"
                        })
                        continue

                    # 执行工具
                    try:
                        result = await self.tools.execute(tc)
                        # 截断过长结果（超过 8000 字符的工具输出对 LLM 来说就是噪音）
                        if len(result) > 8000:
                            result = result[:8000] + f"\n...[截断，原始长度 {len(result)} 字符]"
                        yield self._sse("tool_done", f"执行完成: {tc.function.name}")
                    except Exception as e:
                        result = f"[错误] {str(e)}\n{traceback.format_exc()[:500]}"
                        yield self._sse("tool_error", f"执行失败: {tc.function.name} - {str(e)}")

                    # 把工具调用和结果加入消息历史
                    messages.append({
                        "role": "assistant",
                        "content": None,
                        "tool_calls": [tc]
                    })
                    messages.append({
                        "role": "tool",
                        "tool_call_id": tc.id,
                        "content": result
                    })

                # 回到循环开头，让 AI 看到工具结果继续思考
                continue

            else:
                # AI 返回纯文本 = 任务完成（或不需要工具）
                content = response.content
                yield self._sse("text", content)
                yield self._sse("done", json.dumps({
                    "turns": turn,
                    "tool_calls": total_tool_calls
                }))
                return

        # 超过最大轮次，强制结束
        yield self._sse("warning", f"已达到最大对话轮次({self.max_turns})，任务可能未完全完成。请考虑拆分为更小的子任务。")

    # ── 辅助方法 ──

    def _estimate_tokens(self, messages: list[dict]) -> int:
        """粗暴但实用的 token 估算：1 中文字符 ≈ 1.5 token，1 英文单词 ≈ 1.3 token"""
        total = 0
        for msg in messages:
            content = msg.get("content", "") or ""
            # 简单估算
            chinese_chars = sum(1 for c in content if '一' <= c <= '鿿')
            other_chars = len(content) - chinese_chars
            total += int(chinese_chars * 1.5 + other_chars * 0.3)

            # tool_calls 也占 token
            if msg.get("tool_calls"):
                total += len(json.dumps(msg["tool_calls"])) // 2
        return total

    def _prepare_history(self, history: list[dict], max_messages: int = 30) -> list[dict]:
        """截取最近 N 条历史消息"""
        return history[-max_messages:]

    def _compress_context(self, messages: list[dict]) -> list[dict]:
        """上下文压缩：保留 system prompt + 最近的消息，中间的做摘要"""
        if len(messages) <= 10:
            return messages

        # 保留 system prompt (index 0) + 最后 8 条消息
        system_msg = messages[0]
        recent = messages[-8:]

        # 中间的消息用一段摘要替代
        summary = f"[上下文摘要] 此前已进行了多轮工具调用，包括文件读取、知识库搜索等操作。详细结果已反映在最近的对话中。"
        return [system_msg, {"role": "user", "content": summary}] + recent

    def _safety_check(self, tool_call) -> bool:
        """工具安全检查"""
        func_name = tool_call.function.name
        args = json.loads(tool_call.function.arguments)

        # 文件操作：禁止操作系统关键目录
        if func_name in ("read_file", "write_file"):
            path = args.get("file_path", "").lower()
            forbidden = ["c:\\windows", "c:\\windows\\system32", "/etc/", "/sys/"]
            for fb in forbidden:
                if fb in path:
                    return False

        # 系统命令：禁止危险操作
        if func_name == "run_command":
            cmd = args.get("command", "").lower()
            dangerous = ["format", "del /", "rm -rf", "shutdown", "reg delete"]
            for d in dangerous:
                if d in cmd:
                    return False

        return True

    def _sse(self, event_type: str, data: str) -> str:
        """构造 SSE 格式的事件"""
        return f"event: {event_type}\ndata: {json.dumps(data, ensure_ascii=False)}\n\n"
```

```python
# backend/core/tool_executor.py

import json
import asyncio
from pathlib import Path

class ToolExecutor:
    """工具执行器：根据 AI 的工具调用请求，实际执行操作"""

    def __init__(self, knowledge_base=None, workspace_dir: str = "./workspace"):
        self.kb = knowledge_base
        self.workspace = Path(workspace_dir)
        self.workspace.mkdir(parents=True, exist_ok=True)

        # 工具名 → 处理函数的映射
        self.handlers = {
            "read_file": self._read_file,
            "search_knowledge": self._search_knowledge,
            "generate_word": self._generate_word,
            "generate_ppt": self._generate_ppt,
            "web_search": self._web_search,
            "finish": self._finish,
        }

    async def execute(self, tool_call) -> str:
        """执行单个工具调用，返回字符串结果"""
        func_name = tool_call.function.name
        args = json.loads(tool_call.function.arguments)

        handler = self.handlers.get(func_name)
        if not handler:
            return f"[错误] 未知工具: {func_name}"

        # 设置超时（防止某个工具卡死）
        try:
            result = await asyncio.wait_for(
                handler(args),
                timeout=30.0  # 单个工具最多 30 秒
            )
            return result
        except asyncio.TimeoutError:
            return f"[超时] 工具 {func_name} 执行超过 30 秒，已取消"

    async def _read_file(self, args: dict) -> str:
        path = Path(args["file_path"])
        if not path.exists():
            return f"[错误] 文件不存在: {path}"
        if path.suffix.lower() in ('.pdf', '.pptx', '.docx'):
            # 复杂格式走知识库的 reader
            from knowledge.doc_loader import DocumentLoader
            return DocumentLoader.load(str(path))
        # 纯文本直接读
        content = path.read_text(encoding='utf-8', errors='replace')
        if len(content) > 10000:
            content = content[:10000] + f"\n...[文件过长，已截断。共 {len(content)} 字符]"
        return content

    async def _search_knowledge(self, args: dict) -> str:
        if not self.kb:
            return "[错误] 知识库未初始化"
        query = args["query"]
        top_k = args.get("top_k", 5)
        hits = self.kb.search(query, top_k=top_k)
        if not hits:
            return f"未找到与 '{query}' 相关的内容。建议导入相关资料，或使用网络搜索。"
        result_parts = [f"找到 {len(hits)} 条相关内容：\n"]
        for i, hit in enumerate(hits):
            result_parts.append(
                f"[结果{i+1}] 来源：{hit['source_file']}\n{hit['text'][:500]}\n"
            )
        return '\n'.join(result_parts)

    async def _generate_word(self, args: dict) -> str:
        from tools.word_generator import WordGenerator
        output_path = str(self.workspace / args["output_filename"])
        WordGenerator.create_essay(
            title=args["title"],
            author="",  # 从用户配置读取
            sections=args["sections"],
            output_path=output_path
        )
        return f"文档已生成: {output_path}"

    async def _generate_ppt(self, args: dict) -> str:
        from tools.ppt_generator import PPTGenerator
        output_path = str(self.workspace / args["output_filename"])
        PPTGenerator.create_presentation(
            title=args["title"],
            slides=args["slides"],
            output_path=output_path
        )
        return f"PPT 已生成: {output_path}"

    async def _web_search(self, args: dict) -> str:
        query = args["query"]
        # MVP 阶段用简单的网页搜索
        try:
            # 方案1: 调用搜索引擎 API（Bing/DuckDuckGo）
            # 方案2: 用 LLM 的 web_search 能力（如果模型支持）
            # 这里给一个占位实现
            return f"[网络搜索] 搜索结果暂不可用，请手动搜索: {query}"
        except Exception as e:
            return f"[网络搜索失败] {str(e)}"

    async def _finish(self, args: dict) -> str:
        summary = args.get("summary", "任务完成")
        files = args.get("files_created", [])
        return json.dumps({
            "status": "finished",
            "summary": summary,
            "files": files
        }, ensure_ascii=False)
```

#### 4.8.4 循环终止策略 —— Agent 什么时候停？

这是一个容易被忽略但很关键的问题。Agent 不会自己知道"我该停了"，需要设计终止机制：

| 终止条件 | 触发方式 | 说明 |
|---------|---------|------|
| **自然终止** | AI 返回纯文本（不调工具） | AI 认为任务完成，直接回复用户。最理想的情况 |
| **finish 工具** | AI 主动调用 `finish` 工具 | 显式声明"我做完了"。比自然终止更可靠，强制 AI 做总结 |
| **最大轮次** | `turn >= max_turns`（如 15 轮） | 硬上限，防止死循环。发生时给用户警告 |
| **重复检测** | 连续 3 轮调同一个工具且参数相同 | AI 卡住了。告诉它"你似乎陷入了循环，请换一种方法或直接汇报目前进度" |
| **用户打断** | 前端发送 `/cancel` 信号 | 用户在 UI 点了"停止生成"。Agent 应该优雅退出，保留已有进度 |
| **Token 耗尽** | 上下文超过模型上限（如 128K） | 强制压缩或截断。极端情况下如果压缩后仍然超限，汇报"任务太复杂，请拆分" |

```python
# 增强版循环终止（加到 AgentDispatcher.run 里）
def _detect_loop(self, recent_tool_calls: list) -> bool:
    """检测是否陷入重复调用循环"""
    if len(recent_tool_calls) < 3:
        return False
    last_three = recent_tool_calls[-3:]
    # 检查最近3次调用是否一模一样
    return (last_three[0] == last_three[1] == last_three[2])

def _handle_loop_detected(self):
    """检测到循环时，注入一条提示让 AI 跳出来"""
    return {
        "role": "user",
        "content": "[系统提示] 检测到你连续多次执行了相同的工具调用。"
                   "请换一种方法，或直接汇报你目前收集到的信息和进度。"
    }
```

#### 4.8.5 Streaming 与 Agent 循环的关系

一个常见的困惑：Agent 循环是"等待完整结果 → 再决定下一步"的，那打字机效果怎么办？

```
解决方案：双模 Streaming

模式 1: Agent 思考阶段（有工具调用时）
  前端显示状态栏：
  "🔍 正在搜索知识库..."
  "📄 正在读取课件..."
  "✍️ 正在生成文档..."
  （这一阶段不流式推送 LLM 输出，因为 LLM 输出的只是 tool_call 的 JSON）

模式 2: Agent 最终回复阶段（无工具调用时）
  前端切换到聊天模式：
  "好的，我已经为你生成了复习笔记，主要内容包括..."
  （这一阶段开启 SSE 流式，逐字输出，就跟普通 ChatGPT 一样）
```

#### 4.8.7 进阶：ReAct 模式 vs Tool-only 模式

上面的循环本质是 **Tool-only**：AI 想调工具就调，不调就结束。还有一种更强的模式叫 **ReAct**（Reasoning + Acting），让 AI 在每个回合先"说出自己的思考"再行动：

```
Tool-only:
  AI: [调 search_knowledge("二叉树")]

ReAct:
  AI: "我需要理解用户问的是二叉树的哪种遍历方式。从对话来看应该是中序遍历。
       但课件里可能有前序、中序、后序三种，我应该先搜索课件确认完整内容。
       然后根据内容给出准确答案。"
  AI: [调 search_knowledge("二叉树 遍历")]  ← 工具调用前有了"思考链"
```

ReAct 的优势是**思考过程可见**（对 Debug 非常友好，用户也更有安全感），代价是**多消耗 token**（思考文本也占上下文）。

**建议路径**：
- MVP 阶段：Tool-only，简单高效
- P2 阶段：加入 ReAct，让 Agent 思考过程可视化（这本身也是个很好的产品体验——用户看到 AI "在想什么"会更信任）

### 4.9 产出质量保障：降 AI 率、查重、正确性验证

> 用户用你的工具写论文，最怕两件事：① 老师一眼看出是 AI 写的（AI 率过高）；② 内容有错但看不出来（幻觉）。这一节专门解决这两个问题。

#### 4.9.1 先搞清楚：AI 检测的原理

高校查 AI 率用的工具（知网 AIGC 检测、GPTZero、Turnitin 等）核心思路：

```
检测维度                    AI 生成文本的典型特征
─────────                  ────────────────────
词汇多样性                  过于均匀的词汇分布，缺乏人类写作的"突发低频词"
句式规律性                  每句话长度接近、结构雷同（主谓宾整齐排列）
转折/连接词模式             "首先...其次...最后..."、"总而言之"出现频率异常高
信息密度                    每个句子都"言之有物"，缺少口语化的填充和跳跃
思维链痕迹                  "这可以从以下几个方面来分析"、"综上所述"——典型的 LLM 思维框架
温度/随机性                 人类写作有情绪波动，AI 写作从头到尾同一种语气
```

**关键认知**：AI 率检测不是检测"内容是不是真的"（那是事实核查），而是检测"文本的概率分布像不像 AI"。所以降低 AI 率的本质是**让文本的统计特征更接近人类写作**。

#### 4.9.2 降 AI 率的实战策略（按效果排序）

##### 策略 1：分角色分段生成（最有效）⭐⭐⭐⭐⭐

不要一次性让 AI 生成整篇论文。每部分用不同的 System Prompt：

```python
# backend/quality/ai_rate_reducer.py

class AIRateReducer:
    """AI 率降低策略集"""

    # 不同章节用不同的"人设"和"风格"
    SECTION_STRATEGIES = {
        "摘要": {
            "style": "精炼简洁的学术摘要风格，用词严谨但不僵硬。控制在200-300字。",
            "temperature": 0.7,  # 略高温度引入自然的用词变化
            "avoid_phrases": ["本文通过", "研究结果表明", "综上所述"],
        },
        "引言": {
            "style": "像一个正在做文献调研的研究生，自然地引出问题。可以加入你对这个领域的个人困惑和思考动机。语气介于正式和非正式之间。",
            "temperature": 0.8,
            "avoid_phrases": ["随着...的发展", "在当今...背景下", "具有重要的理论意义和实践意义"],
        },
        "文献综述": {
            "style": "像一个读了20篇论文后做笔记的学生。归纳、对比、找矛盾。多用'某某认为...但另一个学者指出...'这种对比结构。",
            "temperature": 0.7,
            "avoid_phrases": ["国内外学者对此进行了大量研究", "目前学术界对...尚无定论"],
        },
        "研究方法": {
            "style": "技术文档风格，客观精确。这一部分允许 AI 味重一点——方法论本身就应该是结构清晰的。",
            "temperature": 0.5,
        },
        "结果与讨论": {
            "style": "像一个在实验室里发现意外结果的学生。对数据有惊讶、有解释、有不确定。加入'有趣的是...'、'出乎意料的是...'这类表达。",
            "temperature": 0.8,
            "avoid_phrases": ["数据表明", "统计结果显示"],
        },
        "结论": {
            "style": "像一个做完项目后的回顾。真诚地总结收获，也承认局限。不要用'本研究具有重要的理论价值和实践意义'这种套话。",
            "temperature": 0.7,
            "avoid_phrases": ["本研究创新性地", "为...领域做出了贡献"],
        },
    }

    @classmethod
    def get_prompt_for_section(cls, section_name: str) -> str:
        strategy = cls.SECTION_STRATEGIES.get(section_name, {})
        style = strategy.get("style", "")
        avoid = strategy.get("avoid_phrases", [])

        prompt = f"""
写作风格要求：{style}

## 禁止使用的套话
{chr(10).join(f'- {p}' for p in avoid)}

## 额外要求
- 偶尔使用短句（5-10字），打破长句节奏
- 可以偶尔写半句然后破折号转折——像人的思维跳跃
- 适当使用口语化的学术表达（"说白了就是..."、"打个比方..."）
- 不要在每段开头都用相同的句式结构
"""
        return prompt

    @classmethod
    def get_temperature_for_section(cls, section_name: str) -> float:
        return cls.SECTION_STRATEGIES.get(section_name, {}).get("temperature", 0.7)
```

##### 策略 2：注入个人素材（决定性的）⭐⭐⭐⭐⭐

纯靠 AI 生成 = 必然被检测。加入**只有你自己有的素材**：

```
用户导入的内容                AI 如何使用
─────────────                ────────────
课程笔记/随堂感想            → 插入到"引言"做个人动机陈述
自己画的草图/示意图          → 在"方法"部分引用
做实验时的原始数据截图       → 在"结果"部分引用具体数值
老师课上的原话               → 在"文献综述"里引用
自己和同学的讨论记录         → 自然穿插在各处的论据中
```

这就是**本地知识库的另一个隐藏价值**——不只是让 AI 参考课件内容，更是让 AI 引用你的个人素材，增加文本的"人类唯一性"。

#### 4.9.3 正确性验证：三道防线

AI 写的内容可能"看起来对但其实错"。学生交了有错误的论文比交了 AI 味的论文更惨。需要三道防线：

```
防线 1: 源头约束（Prompt层）
  → System Prompt 里写死："不确定的事实请标注[需核实]"
  → 禁止编造参考文献、数据、实验结论
  → 所有数据和引用必须来自知识库或网络搜索的实际结果

防线 2: 交叉验证（工具层）
  → 对于关键事实断言，自动调用 web_search 找来源
  → 对于计算题/数学题，用 Python 执行计算验证（而非信任 LLM 的计算能力）
  → 对于引用，检查是否真的在知识库里存在

防线 3: 用户确认（交互层）
  → 生成论文后，标注出 AI 不确定的部分（黄色高亮）
  → 参考文献每一项标注来源（知识库/网络搜索/AI推测）
  → 最后给用户一个"请核对以下内容"的 checklist
```

```python
# backend/quality/fact_checker.py

class FactChecker:
    """事实核查器——三道防线实现"""

    @staticmethod
    async def verify_claims(text: str, kb, search_client) -> list[dict]:
        """对文本中的事实性断言进行交叉验证"""
        # 提取文本中的关键断言（数字、引用、定义）
        claims = FactChecker._extract_claims(text)

        results = []
        for claim in claims:
            # 检查知识库
            kb_result = kb.search(claim, top_k=3) if kb else []
            # 检查网络（如果是事实性问题）
            web_result = await search_client.search(claim) if search_client else ""

            results.append({
                "claim": claim,
                "kb_verified": len(kb_result) > 0 and FactChecker._has_support(claim, kb_result),
                "web_verified": bool(web_result),
                "confidence": "high" if (kb_result or web_result) else "low",
                "suggestion": "建议核实" if not (kb_result or web_result) else None,
            })

        return results

    @staticmethod
    def _extract_claims(text: str) -> list[str]:
        """提取文本中需要核实的事实断言"""
        import re
        claims = []

        # 提取数值断言（"XX%的..."、"XX年..."）
        number_patterns = [
            r'\d+%的[一-鿿]+',  # "30%的学生"
            r'\d{4}年[一-鿿]+',  # "2023年..."
        ]
        for pattern in number_patterns:
            claims.extend(re.findall(pattern, text))

        # 提取引用断言（"XX认为..."、"根据XX..."）
        citation_patterns = [
            r'[一-鿿]{2,4}认为',
            r'根据[一-鿿]{2,10}的',
        ]
        for pattern in citation_patterns:
            claims.extend(re.findall(pattern, text))

        return claims[:20]  # 最多检查 20 条

    @staticmethod
    def _has_support(claim: str, kb_results: list) -> bool:
        """检查知识库是否有支持性的证据"""
        for r in kb_results:
            if any(word in r['text'] for word in claim[:10]):
                return True
        return False

    @staticmethod
    async def verify_math(expression: str) -> dict:
        """验证数学计算——用 Python 实际算一遍，不信任 LLM 的答案"""
        import subprocess
        try:
            result = subprocess.run(
                ['python', '-c', expression],
                capture_output=True, text=True, timeout=5
            )
            return {
                "expression": expression,
                "result": result.stdout.strip(),
                "llm_correct": None,  # 需要和 LLM 的答案比较
            }
        except Exception as e:
            return {"expression": expression, "error": str(e)}
```

#### 4.9.4 查重策略

```
降重 ≠ 降AI率。查重查的是"和别人写的一样"，AI率查的是"像AI写的"。

降重策略（AI 辅助）:
1. 同义替换 → 但不是简单的换词，而是换表达角度
   "二叉树是一种数据结构" → "在计算机存储数据的方式中，二叉树是其中一个经典方案"
2. 语序调整 → 主动变被动、前置定语变后置从句
3. 案例替换 → 用知识库里的具体案例替代教科书的标准案例
4. 句式拆分/合并 → 一个长句拆成两短句，或反过来

这些都可以通过 Prompt 控制，不复杂。真正降低查重率的关键：
尽量从用户的知识库（课件、笔记）里取素材，而不是让 LLM 凭空编。
因为这天然造成了"个体差异"——每个人的课件和笔记措辞都不一样。
```

### 4.10 Token 经济性：少轮次出好活

> 你提到的核心矛盾——"跑很多遍烧 token 产出还不好就没人用"——这确实是 Agent 产品最大的坑。以下是怎么让它**尽量一轮过，最多三轮**。

#### 4.10.1 先理解 token 烧在哪

一次典型的"写论文"Agent 会话：

```
轮次    做什么                  token 消耗（估算）
────    ────                   ─────
1       AI 规划 + 搜索知识库     ~3K
2       AI 读课件 + 提取内容     ~5K
3       AI 生成第一章           ~4K
4       AI 说"还需要查一下X"    ~3K
5       AI 搜索X + 生成第二章    ~5K
6       AI 生成第三章           ~4K
7       AI 生成结论             ~3K
8       AI 整合全文 + finish    ~5K
        ────────
        总计                    ~32K token

如果用 DeepSeek（¥1/百万 token）：成本约 ¥0.03
如果用 Claude（$3/百万 token）：成本约 $0.10
```

Token 费用本身极低。**真正的问题不在钱，在"用户体验"**——如果每轮工具调用加上思考时间要 5-8 秒，8 轮就是 40-60 秒，用户等得烦躁。而且轮次越多，AI 越容易在中间跑偏。

#### 4.10.2 核心策略：一次规划，批量执行

对比两种模式：

```
❌ 串行模式（低效，容易跑偏）:
  Turn 1: AI "我需要搜索知识库"  → 等结果
  Turn 2: AI "搜到了，让我读取课件" → 等结果
  Turn 3: AI "读完了，开始生成第一章" → 开始干活
  Turn 4: AI "第一章写完了，写第二章" → ...

✅ 规划先行模式（高效，不易跑偏）:
  Turn 1: AI "我先规划一下：1)搜知识库 2)读课件 3)生成论文"
          → [搜知识库, 读课件]  ← 批量调用
  Turn 2: AI "信息都拿到了，开始生成论文"
          → [generate_word]        ← 一步到位
  Turn 3: AI "完成，检查质量"      ← finish
```

实现方式：**在 System Prompt 里教会 AI 做批量工具调用**：

```python
# 在 Agent System Prompt 中加入这段

BATCH_EXECUTION_PROMPT = """
## 高效工作原则（非常重要）

为了减少用户的等待时间，请遵循以下原则：

1. **先规划，再动手**
   在调用任何工具之前，先在思考中列出完整的"待办清单"。
   如果任务涉及多个步骤，一次性想清楚先后顺序和依赖关系。

2. **能批量的就批量**
   如果你需要同时做几件互不依赖的事情（如：同时搜索知识库 + 读一个文件），
   在一次回复中调用所有工具，而不是逐个等待。

3. **有足够信息就直接产出**
   当你已经有了完成任务所需的所有信息，不要为了"显得谨慎"而额外做无意义的确认。
   直接生成最终文档。

4. **不要"过度工具化"**
   - 能直接回答的问题不要搜索
   - 能一次生成的内容不要分两次
   - 能靠常识判断的不要查资料确认
   每一次不必要的工具调用都是在浪费用户的时间和 token。

5. **3轮原则**
   你应该在 3 轮之内完成绝大多数任务。
   第1轮：规划 + 收集信息
   第2轮：执行生成
   第3轮：检查 + 交付
   如果需要超过3轮，说明任务应该拆分成两个独立的请求。
"""
```

#### 4.10.4 缓存中间结果

如果用户反复问同一份课件的相关问题，不要把课件反复读出来 → embedding → 喂给 AI。第一次加载后就缓存：

```python
# 简单的文件内容缓存
import hashlib
import time

class FileCache:
    """文件读取缓存：避免重复读同一文件 burn token"""

    def __init__(self, ttl: int = 3600):  # 缓存 1 小时
        self.cache = {}  # {file_hash: (content, timestamp)}

    def get(self, file_path: str) -> str | None:
        key = self._hash(file_path)
        if key in self.cache:
            content, timestamp = self.cache[key]
            if time.time() - timestamp < self.ttl:
                return content
        return None

    def set(self, file_path: str, content: str):
        self.cache[self._hash(file_path)] = (content, time.time())

    def _hash(self, path: str) -> str:
        return hashlib.md5(path.encode()).hexdigest()
```

```python
# backend/skills/skill_engine.py

import yaml
from pathlib import Path

class SkillEngine:
    """Skill 加载和执行引擎"""

    def __init__(self, skills_dir: str = "./skills"):
        self.skills_dir = Path(skills_dir)
        self.skills = {}  # name → skill_definition
        self._load_builtin_skills()

    def _load_builtin_skills(self):
        """加载所有内置 Skill"""
        for yaml_file in self.skills_dir.glob("*.yaml"):
            with open(yaml_file, 'r', encoding='utf-8') as f:
                skill = yaml.safe_load(f)
                self.skills[skill['name']] = skill

    def match_skill(self, user_input: str) -> dict | None:
        """根据用户输入匹配合适的 Skill"""
        for name, skill in self.skills.items():
            for trigger in skill.get('triggers', []):
                keywords = trigger.get('keywords', [])
                if any(kw in user_input for kw in keywords):
                    return skill
        return None  # 没有匹配的 Skill，走默认 Agent 循环

    def get_skill_prompt(self, skill: dict, step_id: str = None) -> str:
        """获取某个 Skill 步骤的 System Prompt"""
        if step_id:
            for step in skill['workflow']['steps']:
                if step['id'] == step_id:
                    return step.get('prompt', '')
        # 返回整个 Skill 的概览
        return skill.get('description', '')
```

```
  └── 智谱 ChatGLM API    → 国内直接访问 ✅

第2层：搜索 API
  ├── Bing Search API     → 可访问但有配额限制 ✅
  ├── SerpAPI             → 需要代理 ⚠️
  ├── DuckDuckGo          → 免费但有时不稳定 ⚠️
  └── 国内搜索引擎（百度搜索API）→ 信息质量不如 Google/Bing ❌

第3层：需要代理的外网（GitHub、Google Scholar、arXiv等）
  ├── 方案A: 用户自己配代理 → 设置页填代理地址
  ├── 方案B: 云端服务做中转 → 你的服务器访问外网 → 返回结果给客户端
  └── 方案C: 国内镜像站    → arxiv.org → xxx.arxiv.org 国内镜像
```

**实用实现**：

```python
# backend/tools/web_tools.py

import httpx
import asyncio
from urllib.parse import quote

class WebTools:
    """网络查询工具集——适配国内网络环境

    ⚠️ 重要边界：代理配置仅对 Web 搜索和 GitHub API 请求生效。
    DeepSeek API 走国内直连，不受代理影响。
    这不是"全局代理"，只是"网络搜索的辅助通道"。
    降低实现复杂度——不需要热更新 openai SDK 的 http_client。
    """

    def __init__(self, proxy: str = None):
        self.proxy = proxy  # 用户配置的代理地址，如 http://127.0.0.1:7890

    async def search(self, query: str, engine: str = "auto") -> list[dict]:
        """
        网络搜索。engine 可选:
        - "auto": 自动选择可用的搜索引擎
        - "bing": Bing Search API
        - "duckduckgo": DuckDuckGo（免费，无 API Key）
        - "serpapi": SerpAPI（Google 结果，需代理+Key）
        """
        if engine == "auto":
            # 优先用免费方案
            try:
                return await self._search_duckduckgo(query)
            except:
                try:
                    return await self._search_bing(query)
                except:
                    return [{"title": "搜索不可用",
                             "snippet": "请检查网络连接，或在设置中配置代理",
                             "url": ""}]
        elif engine == "bing":
            return await self._search_bing(query)
        elif engine == "duckduckgo":
            return await self._search_duckduckgo(query)
        else:
            return await self._search_serpapi(query)

    async def _search_duckduckgo(self, query: str) -> list[dict]:
        """DuckDuckGo 搜索——免费，但结果质量一般"""
        # 使用 duckduckgo_search 库
        from duckduckgo_search import DDGS
        results = []
        with DDGS() as ddgs:
            for r in ddgs.text(query, max_results=5):
                results.append({
                    "title": r.get("title", ""),
                    "snippet": r.get("body", ""),
                    "url": r.get("href", ""),
                })
        return results

    async def _search_bing(self, query: str) -> list[dict]:
        """Bing Search API——需要 API Key（免费层每月1000次）"""
        # 免费申请: https://www.microsoft.com/en-us/bing/apis/bing-web-search-api
        api_key = "YOUR_BING_API_KEY"  # 实际从配置读取
        async with httpx.AsyncClient(proxy=self.proxy) as client:
            resp = await client.get(
                "https://api.bing.microsoft.com/v7.0/search",
                params={"q": query, "count": 5},
                headers={"Ocp-Apim-Subscription-Key": api_key}
            )
            data = resp.json()
            results = []
            for r in data.get("webPages", {}).get("value", []):
                results.append({
                    "title": r.get("name", ""),
                    "snippet": r.get("snippet", ""),
                    "url": r.get("url", ""),
                })
            return results

    async def fetch_github(self, repo: str) -> str:
        """获取 GitHub 仓库信息——国内可能需要代理"""
        url = f"https://api.github.com/repos/{repo}"

        # 方案1: 直连 GitHub API
        try:
            async with httpx.AsyncClient(timeout=10) as client:
                resp = await client.get(url)
                if resp.status_code == 200:
                    return resp.text
        except:
            pass

        # 方案2: 通过代理
        if self.proxy:
            try:
                async with httpx.AsyncClient(proxy=self.proxy, timeout=10) as client:
                    resp = await client.get(url)
                    if resp.status_code == 200:
                        return resp.text
            except:
                pass

        # 方案3: 走云端中转
        try:
            async with httpx.AsyncClient(timeout=10) as client:
                resp = await client.get(
                    f"https://your-cloud-server.com/proxy/github/{repo}"
                )
                if resp.status_code == 200:
                    return resp.text
        except:
            pass

        return f"[无法访问] GitHub API 不可达。请检查网络或配置代理。仓库: {repo}"
```

#### 4.11.4 Skill 设计哲学：蒸馏定式，参数化个性

> 你做的 PPT 工作流，效果不错、不会千篇一律——这本身就是 Skill 的最佳素材。把"你的方法论"固定成执行引擎，把"个性化选择"暴露给用户做参数。

##### 核心洞察

```
❌ 错误理解：Skill = "AI 全自动搞定一件事"
   用户: "帮我做个PPT"
   AI: [自己决定风格/配色/结构] → 产出千篇一律的"AI味"PPT
   用户: "还行...但不是我想要的"

✅ 正确理解：Skill = "你的方法论做发动机，用户的选择做方向盘"
   用户: "帮我做个PPT，用'深空月夜'风格，蓝紫配色，内容分4块"
   AI: [按用户指定的参数，在你验证过的工作流里执行] → 产出符合用户审美的PPT
   用户: "对，这就是我想要的感觉"
```

**一句话**：固定的工作流负责"不出错"，开放的参数负责"不无聊"。

##### 案例：把你的 PPT 工作流蒸馏成 Skill

假设你手头已经有一套做 PPT 的方法——比如先用思维导图理清结构，再确定每页的"标题-要点-配图"三要素，再用特定排版规则填充。把它写成 Skill：

##### 个性化参数的收集流程

这是一个关键的产品体验设计——怎么让用户"定制"而不是被 AI "安排"？

```
传统做法（用户被动）:
  AI: "我做了一个PPT，你看看行不行" → 用户: "不太行..."
  AI: "那你说怎么改" → 用户: "我也不知道怎么说..."
  → 死循环

参数化 Skill 的做法（用户主动）:
  AI: "这个PPT给谁看？课业汇报还是比赛路演？"
  用户: "课业汇报"
  AI: "了解了。风格上，你喜欢'深色科技感'、'白色简洁'还是'暖调学术'？"
  用户: "白色简洁吧"
  AI: "好。内容大概分几块？"
  用户: "四块：树的概念、二叉树、遍历算法、哈夫曼树"
  AI: "收到。我先按这些设定生成大纲，你看一眼——"
  [展示大纲]
  用户: "第三块'遍历算法'多分两页吧，前中后序各一页"
  AI: "没问题，调整好了。现在生成完整PPT——"
  [生成，用户拿到的是按自己选择定制的PPT]
```

##### 实现：参数注入到 System Prompt

```python
# backend/skills/parameterized_skill.py

class ParameterizedSkill:
    """参数化 Skill 执行器"""

    def __init__(self, skill_definition: dict):
        self.definition = skill_definition
        self.params = {}  # 用户选择/填写的参数值

    def build_system_prompt(self) -> str:
        """把用户参数注入到工作流的每一步 System Prompt 中"""
        base_prompt = self.definition.get("base_prompt", "")

        # 把参数拼成一段约束
        param_constraints = []
        param_constraints.append("## 用户指定的个性化设置\n")

        visual = self.params.get("visual_style", {})
        if visual:
            param_constraints.append(f"- 视觉风格：{visual.get('name', '')} - {visual.get('description', '')}")
            param_constraints.append(f"  色彩模式：{visual.get('palette', '')}")

        density = self.params.get("content_density", {})
        if density:
            param_constraints.append(f"- 内容密度：{density.get('name', '')} - {density.get('description', '')}")

        structure = self.params.get("structure", "")
        if structure:
            param_constraints.append(f"- 内容划分：{structure}")

        special = self.params.get("special_requirements", "")
        if special:
            param_constraints.append(f"- 特殊要求：{special}")

        param_constraints.append(
            "\n**重要**：以上是用户明确的选择和偏好。"
            "所有生成的内容必须严格遵循这些设置。"
            "这不是建议，是要求。"
        )

        return base_prompt + "\n" + "\n".join(param_constraints)

    def collect_param(self, param_name: str, value: any):
        """收集一个用户参数"""
        param_def = self.definition.get("user_parameters", {}).get(param_name, {})
        if param_def.get("type") == "choice":
            # 验证 value 在 options 内
            valid_ids = [o["id"] for o in param_def.get("options", [])]
            if value not in valid_ids and value != "custom":
                raise ValueError(f"无效选项: {value}，可选: {valid_ids}")
            # 找到完整的 option 对象
            for opt in param_def.get("options", []):
                if opt["id"] == value:
                    self.params[param_name] = opt
                    break
        else:
            self.params[param_name] = value
```

##### 设计原则总结

| 原则 | 说明 |
|------|------|
| **固定的是方法论，开放的是审美** | 工作流保证质量，参数保证个性 |
| **用户做选择，AI 做执行** | AI 给选项（不是给成品），用户拍板（不是提意见） |
| **参数是具象的，不是抽象的** | 不说"你要什么风格"，说"深色科技感还是白色简洁"——降低用户的决策负担 |
| **存档复用 > 每次重来** | 用户调好一套参数后，存为 Profile。下次一键调用 |
| **下降路径清晰** | 如果用户不想选参数（懒），给一个"默认推荐"一键跳过。但不能没有参数入口 |

### 4.12 现实检查：之前没说的硬骨头

> 4.6 节写了 Office 文件操作的 Demo 代码，但 Demo ≠ 能用。这一节诚实说明每个坑的真实深度，以及你作为单人学生开发者能做到的上限。

#### 4.12.1 Office 文件：每一项都是专业公司的攻坚难题

##### PDF 读取的真实难度

之前写的 `pypdf` + `pdfplumber` Demo 在**纯文字单栏 PDF** 上能跑。但学生的课件/论文实际是：

```
真实 PDF 的复杂度层次：

Lv1 ✅ 纯文字单栏论文（arXiv 导出的那种）
     → pypdf 提取成功率 95%+  ← 你之前看到的 Demo 到这就停了

Lv2 ⚠️ 双栏学术论文
     → pdfplumber 能处理大部分，但图表标题、跨栏段落会乱序
     → 提取成功率 ~70%，需要人工校对

Lv3 🔴 图文混排课件（文字+插图+表格+公式+手写批注）
     → 文字提取只能拿到零散的标题和短句
     → 图片里的内容完全丢失
     → 公式变成乱码或空白
     → 这是高校课件的最常见形态
     → 提取成功率 ~30%，基本上不可用

Lv4 💀 扫描版 PDF（整页都是图片，连文字都是扫描的）
     → pypdf 一个字都抽不出来
     → 需要 OCR：PaddleOCR（中文效果最好，免费）或 Tesseract
     → OCR 本身有错误率（中文 ~95-98% 准确率意味着每 100 字有 2-5 个错字）
     → 数学公式 OCR 基本无解（MathPix API 是商业产品，¥0.3/次）
```

**诚实的 MVP 策略**：
- Lv1 PDF 正常支持
- Lv2 PDF 尽力提取 + 在结果里标注"此文档为双栏排版，提取结果可能不完整，建议对照原文"
- Lv3/Lv4 PDF 直接告诉用户"此文档以图片为主，暂不支持提取。建议手动整理要点后发给我"
- **不要承诺能"读任何 PDF"**——连 Adobe Acrobat 的 OCR 都做不到 100%

##### PPT 读取的真实难度

```
老师课件的典型形态：
- 第3页：一张组织结构图（SmartArt，python-pptx 抽不出文字）
- 第7页：一个大表格（能抽出来，但格式全丢）
- 第12页：标题 + 两张截图（截图里的内容 = 零）
- 第18页：动画流程图（每一层都是单独的 shape，顺序取决于动画时间线）
- 讲师备注里有重要考点（这个能抽，但 80% 的老师不留备注）
```

**诚实的 MVP 策略**：
- PPT 只提取**纯文本**（标题 + 文本框 + 表格文字）
- 图片内容、SmartArt、图表标注 → 标注"此页存在图片/图形内容，文字提取不完整"
- `.ppt` 老格式提示用户先在 PowerPoint 里另存为 `.pptx`

##### PPT 生成的真实难度

之前写的 `PPTGenerator` 是最简 Demo——白底 + 标题 + 几个 bullet point。但用户期望的是：

| 用户期望 | Demo 能力 | 差距 |
|---------|----------|------|
| 好看的模板 | 白底黑字 + 几个色块 | 🕳️ 模板设计需要设计师 |
| 图文并茂 | 纯文字 | 🕳️ 图片搜索+插入+裁剪+对齐是一整套系统 |
| 图表自动生成 | 不做 | 🕳️ 需要理解数据 → 选图表类型 → 渲染 |
| 动画/切换效果 | 不做 | 🕳️ python-pptx 支持但极繁琐 |
| 字体统一好看 | 可能乱码 | 🕳️ 中文字体跨平台兼容是玄学 |

**诚实的 MVP 策略**：
- 承认定位：生成的 PPT 是**"内容骨架"**——结构对、要点全、格式整洁
- 用户拿到后需要在 PowerPoint/WPS 里自己调样式、加图片、做美化
- 提供 3~5 套配色模板（用你的 UI 色彩体系），比白底强
- **不承诺"一键做出精美 PPT"**，而是"一键生成内容骨架，你自己润色"

##### Word 排版的地狱

```
你以为的 Word 生成：AI 写文字 → 塞进 docx → 搞定

实际的 Word 生成：
├── 封面（居中、字号层级、下划线、校徽图片位置）😰
├── 中文摘要 + 英文摘要（不同页码格式）😰
├── 目录（需要 Word 内部渲染才能生成页码）💀
├── 正文：
│   ├── 多级标题编号（一、(一) 1. 1.1 (1) ①）😰
│   ├── 图表自动编号（"图1-2"、"表3-1" 交叉引用）💀
│   ├── 数学公式排版（MathType/OMML 格式）💀
│   ├── 三线表格式（学术论文标配）😰
│   └── 参考文献引用上标 [1][2-4] 😰
├── 页眉页脚（奇偶页不同、每章不同）💀
├── 参考文献（GB/T 7714 格式自动生成）😰
└── 致谢页 😊 ← 这个最简单，纯文字居中就行

😊 = 容易  😰 = 需要大量模板代码  💀 = python-docx 基本做不到
```

**诚实的 MVP 策略**：
- 专注做好**正文内容生成**（这是 AI 的强项：内容质量）
- 封面/目录/页眉页脚/交叉引用——这些 python-docx 做不了或代价极大
- 生成后提示用户："正文内容已完成。封面、目录、页眉页脚请使用 Word 内置功能添加（操作指引见帮助页）"
- **目标不是"生成完美论文"，而是"省去 70% 的文字工作"**——剩下的 30% 是 Word 排版，学生自己用学校模板套

#### 4.12.2 Python 打包：PyInstaller 的真正面目

```
之前写的：
  pyinstaller --onefile main.py  → dist/backend.exe (~80-120MB)

没说的事：
├── 第一次打包大概率闪退，没有日志
├── --onefile 模式每次启动要解压到临时目录，冷启动 +3~5 秒
├── chromadb 的 C 扩展（hnswlib）经常打不进去，需要手动 --add-binary
├── 某些包依赖的 DLL 在不同 Windows 版本上行为不同
├── 杀毒软件对 PyInstaller 打包的 exe 误报率很高（尤其是 360）
├── pyinstaller 打出来的 exe 在 Win11 上 SmartScreen 必拦
└── 真正的体积不是 80MB——加上 numpy/pandas/torch 分分钟 300MB+
```

**实际可行方案**：

| 方案 | 体积 | 优点 | 缺点 |
|------|------|------|------|
| PyInstaller --onedir | ~120-200MB | 启动快，调试方便 | 目录散乱，300+ 文件 |
| PyInstaller --onefile | ~80-150MB | 单文件干净 | 启动慢，解压到 %TEMP% |
| Nuitka | ~60-100MB | 编译为 C 再编译为机器码，性能好 | 编译 30-60 分钟，商业版才支持 --onefile |
| **嵌入 Python 运行时** ✅ | ~40-60MB（不含依赖）+ 依赖 pip install | 最灵活，用户可自行 pip install 更新 | 需要用户有网络 |
| **不打包，用安装器** ✅ | 安装包内嵌 Python embeddable | 可控性强 | 制作安装器需要额外学习 NSIS/Inno Setup |

**推荐路线**：
- 开发阶段 → 直接 `python main.py`，不打包
- 内测阶段 → PyInstaller --onedir（能跑就行）
- 正式分发 → Inno Setup 制作安装包，内嵌 Python embeddable + pip 依赖，启动时自动检查依赖完整性

#### 4.12.4 Token 优化：别用 Base64 传图，Tauri Clipboard 方案

之前提到"用户粘贴截图给 AI 看题目"时，默认做法是前端 `canvas.toDataURL()` → Base64 字符串 → 塞进 API 请求。一张 1080p 截图转 Base64 约 **2-4MB 文本**，折算成 token：

```
截图的 token 消耗：
  Base64 编码: ~2,000,000 字符 → ~500,000 token（约 ¥0.50/次 DeepSeek）
  如果用户一天问 20 道题 = ¥10/天 = ¥300/月 ← 完全不可接受
```

**Tauri Clipboard 方案**：不走 API，走系统剪贴板。

```javascript
// 前端：用户截图后，写到系统剪贴板
import { writeImage, readImage } from '@tauri-apps/plugin-clipboard-manager';

// 用户按 Ctrl+V 粘贴截图
document.addEventListener('paste', async (e) => {
    // Tauri 直接读剪贴板的图片二进制
    const base64_image = await readImage();  // 拿到底层二进制
    // 这不会出现在 LLM API 请求里！
});

// 正确做法：截图保存为本地文件 → AI 只收文件路径 → 本地 OCR 预处理
// 流程: 截图 → 保存到 data/screenshots/ → OCR 提取文字 →
//       → 只把文字发给 AI（token 消耗 ~500 token → ~0.0005 元）
```

**完整优化方案**：

| 场景 | 之前做法 | 优化后 | Token 节省 |
|------|---------|--------|-----------|
| 用户粘贴截图问题目 | Base64 传 Vision API | Tauri clipboard → 本地保存 → OCR 提取文字 → 只送文字给 AI | **99.9%** |
| 用户"帮我看看这张图" | Base64 → Vision API | 本地保存 → 缩略图判断是否需要 Vision API → 只在必要时用 | **90%+** |
| 用户传文件 | Base64 整个文件 | 本地读文件路径 → 后端直接读 | **100%** |
| 连续对话重复引用同一份课件 | 每次都把课件内容放 context | 第一次后缓存文件摘要（仅存 embedding 向量和章节概要） | **80%+** |

```python
# backend/tools/ocr_tool.py —— 本地 OCR 预处理，避免 Vision API 调用

class OCRPreprocessor:
    """截图 → OCR → 纯文本，避免用 Vision API 烧 token"""

    @staticmethod
    async def extract_text_from_screenshot(image_path: str) -> str:
        """从截图中提取文字，不调用 AI"""
        # 方案1: PaddleOCR（中文最强，免费）
        from paddleocr import PaddleOCR
        ocr = PaddleOCR(lang='ch')
        result = ocr.ocr(image_path)
        texts = []
        for line in result[0]:
            texts.append(line[1][0])  # line[1][0] 是识别出的文字
        return '\n'.join(texts)

        # 方案2: Windows 内置 OCR API（通过 Tauri 调，零依赖）
        # 在 Tauri 端用 Windows.Media.Ocr 直接识别
        # 然后通过 HTTP 把纯文本传给 Python 后端
```

#### 4.12.5 Tauri 原生能力偷 Token

Tauri 提供了一大堆 JS API，可以在前端直接调用系统能力，**完全不经过 AI API**：

```javascript
// 这些操作都不消耗 token：

// 1. 文件对话框（不用 base64 传文件）
import { open } from '@tauri-apps/plugin-dialog';
const file = await open({ filters: [{ name: '文档', extensions: ['pdf', 'docx'] }] });
// file 是本地路径字符串 → 传给后端 → 后端直接读文件

// 2. 剪贴板（不用 base64 传截图）
import { readText, readImage } from '@tauri-apps/plugin-clipboard-manager';
const text = await readText();  // 用户复制的文字直接拿到

// 3. 系统通知（不需要 AI 生成通知内容，前端自己发）
import { sendNotification } from '@tauri-apps/plugin-notification';
sendNotification('PPT 生成完成！');

// 4. 打开文件/文件夹（生成的文档一键打开）
import { open } from '@tauri-apps/plugin-shell';
await open('C:\\Users\\xxx\\workspace\\论文.docx');  // 用系统默认程序打开

// 5. 文件系统操作（读/写本地文件，AI 只需要知道路径）
import { readTextFile, writeTextFile } from '@tauri-apps/plugin-fs';
const content = await readTextFile('C:\\...\\笔记.txt');
```

**核心原则**：AI 只处理"需要智能"的部分（理解意图、生成内容、检索匹配）。其余所有"不需要智能"的操作（读文件、存截图、开文档、发通知）全部走 Tauri 原生 API，**AI 只收到最小必要的文本**。

## 8. 可行性评估

### 8.1 技术可行性：⭐⭐⭐⭐（4/5）

- **Python 后端**：你最熟悉的技术栈，FastAPI + Agent 调度在技术上完全可行，社区有大量参考（LangChain、AutoGPT、OpenAI SDK）
- **前端**：HTML/CSS/JS 是你的已有技能，Chat UI 复杂度不高
- **Tauri 外壳**：只需写 ~200 行 Rust。Tauri 脚手架自动生成 90% 代码。对于一个有 Python/JS 基础的大一学生来说**完全可行**
- **核心风险不是"做不出来"，而是"前期对 Rust 的陌生感可能导致心理阻力"**

### 8.2 时间可行性：⭐⭐⭐（3/5）

- 你是大一学生，课业压力相对可控，寒暑假有整块时间
- 1~2 年的规划合理，对标 RemindAI 这类项目也是单人开发 1~2 年的产出
- **风险**：如果 Rust/Tauri 阶段卡太久（>2个月），果断退回到 Electron

### 8.3 产品可行性：⭐⭐⭐⭐（4/5）

- "大学生 AI 助手"这个需求是真实存在的（论文、PPT、习题都是高频刚需）
- 现有产品（ChatGPT、Claude、Kimi、豆包等）要么是纯 Web，要么面向泛用户，没有一个专门为"大学生学习场景"深度优化的桌面客户端
- 你的差异化方向是正确的

### 8.4 商业可行性：⭐⭐（2/5）

- 不指望盈利是对的——这个赛道很难赚钱
- 但"覆盖服务器 + API 成本"的目标非常务实，在 20~50 个付费用户时即可实现
- 最大价值是**你的个人成长**：全栈项目经验、Rust 入门、AI 应用开发经验

### 8.5 总体结论

> **可以做。而且应该做。**
>
> 不是为了颠覆行业，而是为了：1) 锻炼自己的技术能力；2) 给身边的同学提供一个真正好用的工具；3) 证明"用 AI 辅助薄弱语言开发"是可行的。
>
> 最大的风险不是失败，而是前期因为对陌生技术栈的恐惧而放弃。所以**阶段 0 集中 2 个月攻壳**——如果 Tauri 确实跑不通，果断切 Electron。不要在一个技术点上死磕超过 2 个月。

### 10.3 给你的个人建议

1. **不要闭门造车**：每个阶段完成后都找同学试用，有些问题你自己发现不了但用户 5 分钟就会发现。

2. **阶段性开源**：阶段 1 MVP 完成后就可以把代码放到 GitHub 上。开源带来关注、反馈、甚至贡献者。RemindAI 也是开源的。

3. **写作记录过程**：用博客/公众号/视频记录你的开发过程。"大一学生从零造桌面 AI Agent"这件事本身就很有故事性，会给你带来意想不到的资源。

4. **不要完美主义**：MVP 一定是粗糙的——界面丑、功能少、Bug 多，这都很正常。先做出来，让人用上，再迭代。Perfection is the enemy of done.

5. **珍惜大学时光**：大一就能想到做这样的项目，你的视野已经超过绝大多数同龄人了。这个项目不管最终做成什么样，过程中学到的东西都会是你未来最宝贵的资产。

6. **关于 Rust**：阶段 0 只需要写 ~200 行 Rust（Tauri 自动生成了 90%）。如果 2 个月还没跑通全链路，果断切到 Electron——壳代码全部用你熟悉的 JS 写。**技术选型服务于产品目标，而不是反过来。** C++/CEF 已经确认不是最优方案，不要在它身上浪费时间。
