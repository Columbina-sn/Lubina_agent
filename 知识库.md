## 4. 文件树系统

### 4.1 文件夹选择

用 Tauri 对话框插件打开原生文件夹选择器：

```js
// bridge.js
async function openFolderDialog() {
  return await _invoke('plugin:dialog|open', {
    options: { directory: true, multiple: false, title: '选择文件夹作为工作区' }
  });
}
```

**关键点**：参数必须包一层 `{ options: {...} }`，不能直接传 `{ directory: true }`。这是 Tauri v2 dialog 插件的要求。

### 4.2 真实目录读取

```rust
// lib.rs — 自定义 Rust 命令，绕过 tauri-plugin-fs 的 scope 限制
#[tauri::command]
fn list_dir(path: String) -> Result<Vec<DirEntry>, String> {
    let dir = Path::new(&path);
    let entries = fs::read_dir(dir)?;
    // 收集、排序（目录优先 + 字母序）
    result.sort_by(|a, b| b.is_dir.cmp(&a.is_dir)
        .then(a.name.to_lowercase().cmp(&b.name.to_lowercase())));
    Ok(result)
}
```

**为什么不用 tauri-plugin-fs 的 readDir**：它有 scope 限制，默认只让读应用自身目录。用户选的桌面文件夹会报 `forbidden path`。

### 4.3 懒加载

大型目录不一次性全读——目录节点标 `lazy: true`，点击展开时才 `listDir`：

```js
if (isExpanding && node.lazy) {
  const entries = await Bridge.listDir(node.path);
  node.children = entries.map(...);
  node.lazy = false;
}
```

**为什么**：如果用户选了 `C:\`，一次性递归读取所有子目录会非常慢（数万个文件）甚至卡死。懒加载只读当前层，展开时才读下一层。

### 4.4 全局工作区根目录

```js
// file-explorer.js — 打开文件夹后
window.__lubina_workspace_root = rootPath;
```

挂到 `window` 上而不是在模块内部，是为了让 chat.js、app.js 等其他模块也能直接读到，不需要通过 FileExplorer 中转。

### 4.5 snake_case 陷阱

Rust 返回的 JSON 字段是 `is_dir`（snake_case），不是 JS 习惯的 `isDir`（camelCase）。bridge.js 的 `listDir()` 做了转换：

```js
e => ({ name: e.name, path: e.path, isDir: e.is_dir })
```

## 5. 工作区安全模型

### 5.1 为什么需要

Agent（Plan/Auto 模式）会操作文件——读、写、创建、删除。如果没有任何限制，恶意 prompt 可能让 Agent 读到 `C:\Users\xxx\AppData\` 下的敏感文件。

### 5.2 两层防护

**前端（chat.js）**——发送前校验：
```js
if (chatMode === 'plan' || chatMode === 'agent') {
  if (!window.__lubina_workspace_root) {
    _showWorkspaceRequiredModal(); // 弹窗引导选文件夹
    return;
  }
}
```

**后端（将来）**——路径安全函数（app.js 已提供）：
```js
function isPathInWorkspace(filePath) {
  const root = window.__lubina_workspace_root;
  if (!root) return false;
  // 统一分隔符 + 小写比较（Windows 路径不区分大小写）
  const n = filePath.replace(/\\/g, '/').toLowerCase();
  const r = root.replace(/\\/g, '/').toLowerCase();
  // 禁止路径穿越攻击
  if (n.includes('..')) return false;
  return n.startsWith(r);
}
```

`..` 是路径穿越的经典手段（`/workspace/../secret.txt` → 逃逸到 `/secret.txt`），直接禁止最简单安全。

## 6. CodeMirror 6 编辑器

### 6.1 为什么用 CM6

| 问题 | textarea | CM6 |
|------|----------|-----|
| 行号对齐 | 手动 div + scroll 同步，滚动时易错位 | 共享 scrollDOM，永不偏移 |
| 自动换行行号 | 估算字符宽度计算视觉行，不准 | 原生支持，精准 |
| 语法高亮 | 需手写正则（没做） | 按扩展名加载专业语言包 |
| 大文件 | >100KB 卡 | Viewport 只渲染可见行，流畅 |

### 6.2 安装方式：npm（不是 CDN）

```bash
npm install codemirror @codemirror/lang-javascript @codemirror/lang-python ...
```

**为什么不用 CDN**：尝试了 5 种 CDN 方案全部失败：
- esm.sh 的 `codemirror` 和 `@codemirror/view` 是分开发送的，内部版本不一致 → `undefined extension` 错误
- Facet 类型的 `.of()` 方法在某些导入方式下丢失 → `Facet.of is not a function`
- Vite 拦截裸标识符 `import('codemirror')` 尝试本地解析
- importmap 方案与 Vite 冲突

npm 安装后 Vite 从本地 node_modules 解析，lock 文件保证所有子包版本一致，零问题。

### 6.3 核心创建代码

```js
import { EditorView, basicSetup, keymap } from 'codemirror';

const extensions = [
  // basicSetup 含：行号、高亮活动行、特殊字符、选中绘制、历史、缩进、括号匹配、搜索快捷键
  ...[].concat(basicSetup),

  // Ctrl+S 保存
  keymap.of([{ key: 'Mod-s', run: saveFile, preventDefault: true }]),

  // Ctrl+C 复制（显式绑定，弥补 Tauri webview 原生复制不稳定）
  keymap.of([{ key: 'Mod-c', run: copyText, preventDefault: true }]),

  // 文件修改标记（带初始化保护）
  EditorView.updateListener.of(u => {
    if (u.docChanged && _cmSettled) {  // _cmSettled 防止初始化误触发
      file.modified = true;
    }
  }),
];

const view = new EditorView({ doc: 'Hello', extensions, parent: domEl });
```

### 6.4 初始化保护（_cmSettled 计时器）

**问题**：CM6 创建编辑器后，语言扩展、缩进规范化等会触发多次 `docChanged` 事件。如果用单一的 `initial = true` 标志只跳第一次，剩余的仍会误标记文件为"已修改"。

**解决方案**：用 300ms `setTimeout` 设置 `_cmSettled` 标志，这个窗口期内所有 `docChanged` 一律忽略：

```js
let _cmSettled = false;
setTimeout(() => { _cmSettled = true; }, 300);
// updateListener 中：if (u.docChanged && _cmSettled) { ... }
```

**为什么 300ms**：CM6 初始化通常 50-150ms 内完成，300ms 留了足够余量。即使慢速设备也不会误判。

### 6.5 Ctrl+C 显式绑定

**问题**：Ctrl+A/Z/X/V 有效但 Ctrl+C 不生效。CM6 的 `basicSetup` 自带 `standardKeymap`（包含 `Mod-c`），但 Tauri 2 webview 中原生复制事件可能不稳定。

**解决方案**：显式添加 `Mod-c` 键绑定，用 Clipboard API + execCommand 兜底：

```js
keymap.of([{
  key: 'Mod-c',
  run: (view) => {
    const text = view.state.sliceDoc(view.state.selection.main.from, view.state.selection.main.to);
    if (text) {
      navigator.clipboard.writeText(text).catch(() => {
        // 兜底：textarea + execCommand('copy')
        const ta = document.createElement('textarea');
        ta.value = text; ta.style.position = 'fixed'; ta.style.left = '-9999px';
        document.body.appendChild(ta); ta.select();
        document.execCommand('copy');
        document.body.removeChild(ta);
      });
    }
    return true;  // 阻止默认，由我们处理
  }
}])
```

### 6.6 Compartment 是什么（历史参考）

Compartment 是 CM6 官方的"可替换扩展槽位"，曾用于自动换行切换。自动换行功能已删除，相关代码不存于当前版本。保留此说明供将来参考。

### 6.7 语法高亮

```js
const LANG = {
  js:  async () => { const m = await import('@codemirror/lang-javascript'); return m.javascript(); },
  py:  async () => { const m = await import('@codemirror/lang-python'); return m.python(); },
  md:  async () => { const m = await import('@codemirror/lang-markdown'); return m.markdown(); },
  // ...
};
```

每个语言包的 `import()` 是动态的——只有打开对应类型文件时才加载，不浪费带宽。加载过一次后缓存到 `_langCache`。

### 6.8 主题（深色·浅色双模式）

编辑器主题分成两层，走不同的 API：

| 层 | API | 管什么 |
|----|-----|--------|
| 外壳 | `EditorView.theme()` | 背景色、行号区、光标颜色、选中高亮、字体大小和粗细 |
| 语法高亮 | `HighlightStyle.define()` + `syntaxHighlighting()` | 关键词、字符串、注释、变量名等的颜色 |

**为什么语法高亮不能放在 `EditorView.theme()` 里**：CM6 的语法着色不走 CSS 类名（DOM 上没有 `.cm-keyword` 这种东西），而是通过 tag→style 映射系统。`HighlightStyle.define()` 将语义 tag（`tags.keyword`、`tags.string` 等）映射为颜色，`syntaxHighlighting()` 将其注入编辑器。如果写在 `EditorView.theme()` 的 `.cm-keyword` 选择器里，完全匹配不到 DOM，等于没写。

**深色模式语法配色原则**：所有色值的 HSL 亮度 ≥ 70%，偏白高亮，拒绝暗色调。

| tag | 颜色 | 语法元素 |
|-----|------|---------|
| `tags.keyword` | `#c8c8e0` | if / for / return |
| `tags.string` | `#c8d4c0` | "hello" |
| `tags.comment` | `#6e6e7a` | // 注释（唯一暗色） |
| `tags.variableName` | `#d4d2d8` | 变量名 |
| `tags.tagName` | `#c0c8d8` | `<div>` |
| `tags.heading` | `#d4d0c8` + bold | # 标题 |

## 7. Tauri v2 插件与 IPC

### 7.1 window.__TAURI__ vs __TAURI_INTERNALS__

| 对象 | 哪来的 | 本项目中能用吗 |
|------|--------|---------------|
| `window.__TAURI__` | `@tauri-apps/api` npm 包 | **不能**（没装这个包） |
| `window.__TAURI_INTERNALS__` | Tauri 运行时自动注入 | **能用** |

本项目用 `__TAURI_INTERNALS__.invoke(cmd, args)` 调用所有 Tauri 能力。

### 7.2 插件注册必须三处联动

1. **Cargo.toml**：`tauri-plugin-dialog = "2"`
2. **capabilities/default.json**：`"dialog:allow-open"`
3. **lib.rs**：`.plugin(tauri_plugin_dialog::init())`

而且 `.plugin()` 必须在 Builder 链上调用，不能在 `setup()` 闭包里调用（太晚了，WebView 已初始化）。

### 7.3 打开外部链接（默认浏览器）

```js
// bridge.js
async function openExternal(url) {
  return _safeCall(async () => {
    await _invoke('plugin:shell|open', { path: url });
  }, (() => { window.open(url, '_blank'); })());
}
```

**使用场景**：设置→关于中的 CodeMirror 6 链接，点击时先弹权限确认框，用户同意后调用此函数在默认浏览器中打开。

**为什么不用 `window.open`**：Tauri WebView 中 `window.open` 不保证打开系统默认浏览器，可能在新 WebView 窗口中打开。`plugin:shell|open` 明确使用操作系统默认程序。

**前置条件**：`capabilities/default.json` 必须包含 `"shell:allow-open"` 权限，否则 IPC 调用直接静默失败。这是 Tauri v2 的安全模型——每个插件能力都需要显式声明。

### 7.4 参数格式因插件而异

```js
// dialog 插件 → 需要 { options: {...} } 包装
_invoke('plugin:dialog|open', { options: { directory: true } });

// 自定义 Rust 命令 → 参数直接传
_invoke('list_dir', { path: 'D:/something' });
```

没有统一标准，取决于 Rust 侧的函数签名。

## 8. Rust 自定义命令

### 8.1 为什么需要

`tauri-plugin-fs` 有 scope 安全限制，用户自选的目录（如桌面文件夹）会报 `forbidden path`。写自定义 `#[tauri::command]` 直接用 `std::fs`，完全绕过 scope。

### 8.2 实现

```rust
#[derive(serde::Serialize)]
struct DirEntry { name: String, path: String, is_dir: bool }

#[tauri::command]
fn list_dir(path: String) -> Result<Vec<DirEntry>, String> {
    fs::read_dir(Path::new(&path))
        .map_err(|e| format!("读取目录失败: {}", e))?
        .map(|entry| { ... })
        .collect()
}

#[tauri::command]
fn read_file_content(path: String) -> Result<String, String> {
    fs::read_to_string(&path).map_err(|e| format!("读取文件失败: {}", e))
}
```

### 8.3 注册

```rust
tauri::Builder::default()
    .invoke_handler(tauri::generate_handler![list_dir, read_file_content])
```

`generate_handler!` 是宏——它会为每个命令生成 IPC 路由代码。

### 8.4 前端调用

```js
// 自定义命令没有 plugin: 前缀，直接传命令名
```

### 可缩放布局中的 CSS 单位选择：px vs em vs rem

**什么是 rem**：`rem` = "root em"，相对于 `<html>` 根元素的 `font-size`。本项目通过 `document.documentElement.style.fontSize = (13.5 * scale) + 'px'` 实现字体缩放，所有使用 `rem` 的属性都会随之缩放。

**什么是 em**：`em` 相对于**当前元素**（或继承的）`font-size`。例如 `.tree-node-icon { font-size: 0.9rem; min-width: 1.2em }`，`min-width` 实际 = 1.2 × 0.9rem。

**为什么混合 px 和 rem 会出问题**：

```css
/* ❌ 错误：容器用 px，文字用 rem，字体缩放时文字溢出 */
.tree-node-icon { width: 16px; font-size: 0.9rem; }
/* 字体放大 1.5x 时：文字 = 0.9×13.5×1.5 = 18.2px，但容器只有 16px → 溢出重叠 */

/* ✅ 正确：容器用 em，随文字等比缩放 */
.tree-node-icon { min-width: 1.2em; font-size: 0.9rem; }
/* 字体放大 1.5x 时：容器 = 1.2×18.2 = 21.8px，文字不溢出 */
```

**选择规则**：

| 场景 | 用什么 | 原因 |
|------|--------|------|
| 字体大小 | `rem` | 受根字体缩放控制，"用户调大字体 → 所有文字变大" |
| 图标容器宽度 | `em` | 相对自身 font-size，图标始终不被截断 |
| flex gap / padding | `em` | 随字体缩放保持比例 |
| 边框粗细 | `px` | 通常不需要缩放（1px 边框在各字号下都合适） |
| 固定布局（如侧边栏宽度） | `px` 或 `rem` | 取决于设计意图 |

**SVG 图标缩放**：SVG 有 `width`/`height` 属性（如 `width="14"`），在缩放布局中需要用 CSS 覆盖：
```css
.tree-node-icon svg { width: 1em; height: 1em; }
```
这样 SVG 图标随父元素的 `font-size` 等比缩放。

## 13. 发送拦截与输入框保护

### 13.1 问题

Plan/Auto 模式下未选择工作区时，系统弹窗拦截发送但输入框已被清空。原因是 `app.js` 调用 `Chat.sendMessage(v)` 后无条件执行 `inp.value = ''`——即使 `sendMessage` 因检查提前 return。

```js
// ❌ app.js 旧代码：调用方无条件清空
if (v) { Chat.sendMessage(v); inp.value = ''; }
// sendMessage 可能提前 return，但 inp 已被清空
```

### 13.2 正确模式

输入框清空放在动作函数内部，在所有验证检查通过之后：

```js
// chat.js sendMessage() 新代码
async function sendMessage(content) {
  if (isGenerating || !content?.trim()) return;   // 提前 return，不清空
  if (!chatModelProviderId) { ...; return; }       // 提前 return，不清空
  if (!workspaceRoot) { showModal(); return; }      // 提前 return，不清空

  // === 所有检查通过，确定要发送 ===
  const inp = document.getElementById('chatInput');
  if (inp) { inp.value = ''; inp.style.height = 'auto'; }
  // 继续发送...
}
```

**为什么这样设计**：清空输入框 = "消息已被接受"。将来加余额检查等拦截时只需在 `sendMessage` 中 return，不改调用方。

## 14. 文件夹信任安全检查

### 为什么

Agent 可以读写文件。用户首次打开文件夹时弹确认窗，明确告知 AI 将能操作此文件夹。

### 实现 (`file-explorer.js`)

```js
// 信任列表存 localStorage，JSON 数组
function _getTrustedFolders() {
  try { return JSON.parse(localStorage.getItem('lubina_trusted_folders') || '[]'); }
  catch (_) { return []; }
}

function _normPath(p) { return (p || '').replace(/\\/g, '/').toLowerCase(); }

function _checkFolderTrust(folderPath) {
  return new Promise((resolve) => {
    const trusted = _getTrustedFolders();
    const np = _normPath(folderPath);
    // 父目录或子目录已在信任列表中 → 自动通过
    if (trusted.some(t => np.startsWith(_normPath(t)) || _normPath(t).startsWith(np))) {
      resolve(true); return;
    }
    // 弹安全确认窗 → 用户选"信任"→ 加入列表, resolve(true)
    //                用户选"退出"→ resolve(false) → 清空工作区
  });
}
```

**关键设计**：
- `startsWith` 双向检查：信任父目录→子目录自动受信；反之亦然
- 用户退出时清空 `rootPath` + `__lubina_workspace_root` + `treeData`，回到空状态
- `openFolder()` 和 `setWorkspace()` 都走同一检查流程

## 15. 知识库（SQL + 向量双存储架构）

### 15.1 核心理念

每条知识存两份互为绑定的记录：

| 存储 | 技术 | 面向谁 | 用途 |
|------|------|--------|------|
| `knowledge_items` 表 | SQLite | 用户 | 结构化信息：标题、内容、来源文件、创建时间、"显式/隐藏"标记 |
| `vec_chunks` 表（将来） | sqlite-vec 扩展 | AI | 同一条文本分块 + embedding 向量，语义相似度检索 |

两者通过 `knowledge_items.id` ↔ `vec_chunks.parent_id` 绑定。

### 15.2 两种检索方式

```
用户搜"二叉树" → knowledgeAPI.listItems() → SQL LIKE '%二叉树%' (grep)
AI 问"树形结构有哪些" → 向量化问题 → 在 vec_chunks 中计算余弦距离 → Top-K chunk (RAG)
```

### 15.3 当前实现（第一期）

**数据库表**：
```sql
CREATE TABLE IF NOT EXISTS knowledge_items (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL DEFAULT '',
    content TEXT NOT NULL DEFAULT '',
    source_file TEXT DEFAULT '',
    is_visible INTEGER DEFAULT 1,   -- 1=显式(AI可读) 0=隐藏(仅存储)
    chunk_count INTEGER DEFAULT 0,  -- 预留：将来向量分块数
    created_at TEXT DEFAULT (datetime('now')),
    updated_at TEXT DEFAULT (datetime('now'))
);
```

**后端路由**：`backend/routers/knowledge.py`，路由前缀 `/api/knowledge`：

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/knowledge/items?visible=1/0` | 列显式/隐藏条目 |
| GET | `/api/knowledge/items/{id}` | 单条详情 |
| POST | `/api/knowledge/upload` | 上传文本文件 |
| PUT | `/api/knowledge/items/{id}` | 编辑标题/内容 |
| PUT | `/api/knowledge/items/{id}/toggle` | 切换显隐 |
| DELETE | `/api/knowledge/items/{id}` | 删除 |

**前端 API 封装**：`api.js` 中的 `knowledgeAPI` 对象，路径必须与后端路由完全匹配。

**前端页面**：`knowledge.js` IIFE 模块，SPA 生命周期，双视图切换 + 文件上传 + 卡片列表 + 详情编辑。AI 生成期间通过 `Chat.isGenerating` 检查锁定所有操作。

### 15.4 常见踩坑：事件监听器重复绑定

**问题**：`_render()` 每次调用时都执行 `addEventListener`，但从未移除旧的监听器。N 次渲染后，一次点击触发 N 次回调。

**典型症状**：
- 点击"删除"弹出一层又一层确认窗（N 个监听器 → N 次 `_deleteItem` → N 个 overlay 堆叠）
- 点击"隐藏"按钮没效果（N 个监听器 → N 次 `_toggleVisibility` → toggle 又被 toggle 回来）

**为什么容易犯**：`innerHTML` 替换会销毁子元素的 DOM，子元素上的 `addEventListener` 也随之消失——但**父容器本身**的监听器不受影响。事件委托（在父容器上监听）如果放在 `_render()` 调用链里，每次 render 都多绑一层。

**正确做法**：

```js
// ✅ 用标志确保委托只绑定一次
let _delegatedBound = false;

function _bindEvents() {
  const el = document.getElementById('knowledgePage');
  if (!el) return;

  if (!_delegatedBound) {
    _delegatedBound = true;
    el.addEventListener('click', (e) => {
      const btn = e.target.closest('button[data-action]');
      if (!btn) return;
      // 处理操作...
    });
  }

  // 子元素的直接绑定可以每次 render 重新做
  // （因为旧 DOM 已被 innerHTML 销毁）
  el.querySelector('#kUploadBtn')?.addEventListener('click', _upload);
}

function destroy() {
  _delegatedBound = false;  // 离开页面时重置
}
```

**总结**：
- 事件委托（父容器 listener）→ 绑定一次，用标志防重复
- 子元素直接绑定 → 每次 render 后重新绑（旧 DOM 已死）
- 可以用 `e.stopPropagation()` 防止事件冒泡到其他处理器
- 关键操作（如弹窗）加防重入标志（`if (_deleting) return`）

### 15.5 常见踩坑：前后端路径不一致 → 404

**问题**：后端 `APIRouter(prefix="/api/knowledge")` 生成的路由是 `/api/knowledge/items`，但前端 `api.get('/knowledge/items')` 少了 `/api` 前缀。后端收到请求后找不到匹配路由，返回 404。

**为什么容易犯**：FastAPI 的 `APIRouter(prefix=...)` 和 `app.include_router()` 是声明式拼接——最终路径 = prefix + 路由装饰器路径。如果只看路由文件的 `@router.get("/items")`，容易漏掉 prefix。

**排查方法**：
1. 看后端路由文件的 `APIRouter(prefix=...)` 参数 → 确认完整前缀
2. 看前端 `api.js` 中请求的路径 → 逐段比对
3. 浏览器 DevTools Network 面板 → 看实际发出的 URL
4. 后端日志 → 看收到的请求路径和返回的状态码

**正确对应**：
```
后端: APIRouter(prefix="/api/knowledge") + @router.get("/items") = GET /api/knowledge/items
前端: api.get('/api/knowledge/items')  ← 必须完整匹配
```

## 16. 聊天系统 v6 — 对话级状态 + 消息队列

### 16.1 为什么需要对话级状态

**v5 问题**：`isGenerating`、`chatLocked`、`abortController` 是模块级全局变量。一个对话在等待 AI 回复时，切到其他对话也无法发送消息——因为 `sendMessage()` 第一行就检查全局 `isGenerating`。

**v6 方案**：把这些状态存到每个 conversation 对象上：

```js
// 每个对话独立持有
conv.isGenerating = false;    // 替代模块级 isGenerating
conv.locked = false;          // 替代模块级 chatLocked
conv.abortController = null;  // 替代模块级 abortController
conv._queue = [];             // 新增：排队消息（不持久化）
```

`_switchConversation` 切换对话时，根据新对话自身的 `isGenerating` 设置输入框状态。对话 A 生成中切到对话 B → B 的输入框正常可用。

### 16.2 消息队列机制

**为什么**：发了一条消息等 AI 回复期间，不能再发消息补充或修正内容。用户想追加内容只能等回复完。

**实现**：

```
sendMessage()
  ├─ conv.isGenerating? ─── YES ──→ 入队（conv._queue.push）
  │                                 UI 追加排队气泡（"排队中…"）
  │                                 不阻塞，return
  │
  └─ NO ──→ 正常发送 → _doStreamSend()
                │
                onDone / onError
                  └─ _flushQueue(conv)
                       ├─ 取出所有排队消息 → 加入 conv.messages
                       ├─ 重建整个消息列表
                       └─ 打包全部上下文 → _doStreamSend()
```

**关键设计**：
- 排队消息仅存于 DOM（`_appendQueuedMessage`），不加入 `conv.messages`，所以 `_renderMessages` 不会显示它们
- `_flushQueue` 将排队消息正式加入后，调用 `_renderMessages` 做全量重建，排队气泡变成正式用户气泡
- 队列不持久化（`_saveToStorage` 排除 `_queue`），关软件即丢弃
- `stopGenerating()` 清空队列，用户点"停止"后排队消息消失
- 排队消息和之前的历史一起发送——AI 看到完整上下文

### 16.3 流式发送逻辑复用

`_doStreamSend(conv, apiMessages, aiMsg)` 从 `sendMessage` 中抽取，包含节流渲染、onDelta/onDone/onError 处理、队列消费触发。正常发送和队列消费共用同一函数，避免重复代码。

### 16.4 错误恢复

`_loadFromStorage()` 从 localStorage 恢复对话时：
1. 为旧版本数据补充缺失字段（`isGenerating: false`, `locked`, `_queue`）
2. 遍历所有消息：若 `assistant` 消息 `content` 为空（崩溃/断网导致只保存了空占位），标记为 error 类型，显示"（未获取到 AI 回复，网络可能已断开）"

### 16.5 destroy 安全退出

```js
function destroy() {
  conversations.forEach(c => {
    if (c.abortController) { c.abortController.abort(); ... }
    c.isGenerating = false;
    c._queue = [];
  });
  _saveToStorage();
}
```

切换页面或关软件时，abort 所有正在生成的对话，清空所有队列，保存最终状态。

## 13. Re-Act 循环与工具调用

### 13.1 是什么

**Re-Act**（Reasoning + Acting）= LLM 不仅输出文本，还能**调用工具**获取外部信息，然后基于工具结果继续推理。这是一个循环：

```
用户消息
  → AI 分析问题：需要工具吗？
    → 不需要：直接回复（循环结束）
    → 需要：输出工具调用 JSON {"tool": "...", "parameters": {...}}
      → 后端执行工具，获取结果
      → 工具结果追加到消息历史
      → AI 再次分析：信息够了吗？
        → 够了：给出最终答案
        → 不够：继续调用工具（最多 5 轮）
```

### 13.2 为什么用 Re-Act

没有工具调用的 AI 只能用训练数据中的知识回答。当用户问"我叫什么名字"、"DeepSeek 最新版本是什么"时：

- 纯模型：编造或说不知道
- Re-Act：**先搜知识库**（找用户个人信息）→ 找不到就**联网搜索** → 查到真实信息再回答

### 13.3 三个工具

| 工具 | 文件 | 输入 | 输出 | 用途 |
|------|------|------|------|------|
| KnowledgeGrepTool | `tools/knowledge_grep.py` | `{"query": "..."}` | 格式化的知识库条目 | 查用户存储的信息 |
| WebSearchTool | `tools/web_search.py` | `{"query": "..."}` | 搜索结果摘要 | 四层降级搜索 |
| WebFetchTool | `tools/web_fetch.py` | `{"url": "..."}` | 网页纯文本 | 读取网页详情 |

### 13.4 搜索优先级（写入 System Prompt）

```
1. 涉及个人信息 → 必须先用 knowledge_grep 搜知识库
2. 知识库无结果 → web_search 联网搜索
3. 需要详细内容 → web_fetch 抓取 URL
```

AI 每轮调用一个工具，收到结果后决定是否继续调用。

### 13.5 工具调用上限

为防止无限循环烧 token，设定 **MAX_TOOL_CALLS = 5**。超过上限后强制让 AI 总结现有信息。

### 13.6 停止行为

用户点击停止时：
- `abort_flag = True` → 当前工具操作如果已经开始则**必须完成**（HTTP 请求发出后不可逆）
- 完成后检查 flag → 不再继续循环 → 发送 `done` 事件
- 这就是"操作已经开始的话要做完本次操作再停止"

### 13.7 SSE 事件类型

后端通过 SSE 流式推送多种事件给前端：

| 事件类型 | 含义 | 前端处理 |
|----------|------|----------|
| `{"type":"thinking"}` | AI 思考中 | 显示"思考中……"动画 |
| `{"type":"tool_start",...}` | 开始执行工具 | 插入 tool_call 气泡 |
| `{"type":"tool_result",...}` | 工具执行完成 | 插入 tool_result 气泡 |
| `{"type":"tool_error",...}` | 工具执行出错 | 显示错误信息 |
| `{"type":"delta","content":"..."}` | AI 文本片段 | 追加到 Markdown 气泡 |
| `{"type":"done"}` / `[DONE]` | 对话结束 | 停止流式光标 |

### 13.8 WebSearchTool 四层降级

```python
async def web_search(query: str) -> str:
    # Layer 1: 百度 AI 搜索 API（千帆平台）
    if BAIDU_AI_SEARCH_KEY:
        result = await _baidu_search(query)  # HTTP 429/403 → 降级
        if result: return result

    # Layer 2: Tavily Search API
    if TAVILY_API_KEY:
        result = await _tavily_search(query)  # HTTP 429 → 降级
        if result: return result

    # Layer 3: 预留（服务器中转）
    # Layer 4: 诚实降级
    return "搜索服务暂时不可用，请使用模型的离线知识回答"
```

每层先尝试调用，如果返回 429/403/402（额度耗尽）则捕获并降级到下一层。

## 14. 知识库 AI 提取 + 去重

### 14.1 是什么

用户上传文件后，不是直接存储原始内容，而是：
1. 调用 AI 提取**结构化信息**（每条有 category、content、keywords）
2. 对提取结果做**两层去重**
3. 存入 `knowledge_infos` 表

### 14.2 AI 提取提示词

AI 被要求以 JSON 数组返回提取的信息：
```json
[
  {"category": "个人信息", "content": "用户就读于清华大学", "keywords": ["清华大学", "学生"]}
]
```

重点是提取"AI 模型训练数据中可能不包含的个人信息"。

### 14.3 两层去重

**Layer 1 — 精确去重**：
```python
content_hash = hashlib.sha256(content.encode()).hexdigest()
# SELECT * FROM knowledge_infos WHERE content_hash = ?
# 完全相同 → 跳过
```

**Layer 2 — 模糊去重**：
```python
ratio = difflib.SequenceMatcher(None, new_content, existing_content).ratio()
# ratio >= 0.85 → 合并关键词（set union），更新旧条目
```

### 14.4 为什么这样设计

- 哈希去重：O(1) 查重，处理完全相同的条目
- 模糊去重：difflib 是标准库，零依赖，处理"意思相同但表述略有差异"的条目
- 关键词合并：新旧条目的关键词做集合合并，信息不丢失

### 14.5 knowledge_infos 表结构

```sql
CREATE TABLE knowledge_infos (
    id TEXT PRIMARY KEY,
    category TEXT,       -- 信息类别
    content TEXT,        -- 信息内容
    keywords TEXT,       -- JSON 数组字符串
    source_file TEXT,    -- 来源文件名
    content_hash TEXT,   -- SHA256 哈希
    is_visible INTEGER,  -- 1=显式 0=隐藏
    created_at TEXT,
    updated_at TEXT
);
```
