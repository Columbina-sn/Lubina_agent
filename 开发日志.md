### Bug 修复 — 工具调用链路多项修复

今天修复了以下问题：

**1. `tool_error` 事件被误判为 HTTP 错误**
`tool_error` SSE 事件的 `error` 字段被 `json.error` 检查拦截，导致流被终止 → `json.type` 事件分发移到 `json.error` 检查之前。

**2. AI 工具调用 JSON 泄露到聊天**
AI 返回的工具调用 JSON 在解析前就被流式推给前端 → 改为缓冲模式：先收集完整响应，判断是否为工具调用，仅最终回复才推送 delta。

**3. `role: "tool"` 不兼容 DeepSeek**
OpenAI 协议要求 `role: "tool"` 前必须有带 `tool_calls` 的 assistant 消息 → 改用 `role: "user"` 注入工具结果，纯文本方案。

**4. 双重 `[DONE]` 标记**
`event_generator` 在 done 事件 break 后再次 yield `[DONE]` → 添加 `done_sent` 标志防重复。

**5. Grep 搜索命中率低**
- 存储：AI 提取提示词改为生成**概念/上位词**（如"学校""大学"而非"清华大学"），避免关键词与 content 重复
- 检索：`knowledge_grep` 改为 SQL LIKE 粗筛 + difflib 加权精排（content×2 + keywords×1.5 + category×1 + 精确命中 bonus）
- 调用：System Prompt 教 AI 传多个空格分隔的同义词

**6. 工具调用兜底不足**
- `_safe_str()` 处理 AI 可能输出的 None/列表/数字等奇怪类型
- `_try_parse_json()` 自动修复单引号、尾部逗号等常见 AI JSON 错误
- `_execute_tool` 接受 `q`/`search`/`link`/`address` 等备选参数名，自动补 `https://`

**7. 工具气泡不持久化**
`_saveToStorage` 只保存 `role/type/content`，遗漏 `tool/toolDetail/toolHuman/toolResult` → 补全四个字段。

**8. Markdown 不渲染**
`marked.min.js` 路径错误（新版 marked 在 `lib/marked.umd.js`）→ 脚本 404，标记永远 undefined，`\n` 被 HTML 吞掉 → 修复路径。

### 核心架构 — Re-Act 循环 + 工具调用（重大升级）

将 Lubina 从"直通代理"升级为具备 Re-Act 循环和工具调用能力的智能 Agent。

**原理**：
- 每条用户消息不再直传 AI，而是先拼接 System Prompt（Lubina 身份 + 三个工具定义 + 搜索优先级规则）
- AI 自行判断：简单闲聊直接回复，需要外部信息则输出工具调用 JSON
- 后端解析 JSON → 执行对应工具 → 结果追加到消息历史 → 继续循环
- 最多 5 轮工具调用后强制总结
- 全部通过 SSE 流式推送给前端（`tool_start` / `tool_result` / `delta` / `done` 等事件）

**三个工具**：
| 工具 | 文件 | 功能 |
|------|------|------|
| KnowledgeGrepTool | `tools/knowledge_grep.py` | 搜索知识库 SQLite |
| WebSearchTool | `tools/web_search.py` | 四层降级：百度AI搜索 → Tavily → 预留 → 离线降级 |
| WebFetchTool | `tools/web_fetch.py` | HTTP GET + HTML→纯文本提取 |

**搜索优先级**（写入 System Prompt）：
1. 涉及用户个人信息 → **必须先用 knowledge_grep 搜索知识库**
2. 知识库无结果或需实时信息 → web_search 联网搜索
3. 需阅读详细内容 → web_fetch 抓取

**降级策略**（每层先检测额度再使用）：
- 百度 AI 搜索：HTTP 429/403 → 降级到 Tavily
- Tavily：HTTP 429 → 降级到离线
- 离线：诚实告知用户"搜索服务暂时不可用"

**新增/修改文件**：
- **新**: `backend/services/react_loop.py`, `backend/tools/*.py`
- **重写**: `backend/routers/chat.py`
- **改**: `src/js/chat.js`（SSE 事件分发 + 工具气泡渲染）

### 聊天 — v6 重构：对话级状态 + 错误恢复 + 消息队列

**三合一重构**，解决 `chat.js` 架构层面的三个问题：

**问题 1：无 AI 回复时缺少错误提示**

断网 / 中途关软件 → 重开后对话里只有空 AI 气泡，没有错误标记。

修复：`_loadFromStorage()` 中遍历所有对话消息，若 `assistant` 消息 `content` 为空（崩溃时保存的不完整状态），将其 `type` 改为 `'error'`，`content` 设为 `（未获取到 AI 回复，网络可能已断开）`。

**问题 2：全局 isGenerating 锁死所有对话**

原来 `isGenerating`、`chatLocked`、`abortController` 是模块级全局变量。对话 A 生成时切到对话 B，也发不了消息。

修复：三个变量移入每个 conversation 对象自身（`conv.isGenerating`、`conv.locked`、`conv.abortController`）。`_switchConversation` 切换对话时恢复新对话的输入状态。`Chat.isGenerating` getter 现在读取活跃对话：

```js
get isGenerating() {
  const conv = _getActiveConversation();
  return conv ? conv.isGenerating : false;
}
```

**问题 3：同一对话内无法补充消息**

发了一条消息等 AI 回复期间，不能追加/修正内容。

修复：新增消息队列机制 —
- `sendMessage()` 分两路：`conv.isGenerating` 时走队列路径（不阻塞），否则走正常发送
- 排队消息通过 `_appendQueuedMessage()` 追加到 UI 末尾，显示"排队中…"标记（仅视觉，不加入 `conv.messages`）
- `onDone` / `onError` 回调末尾调用 `_flushQueue()`：取出所有排队消息 → 正式加入 `conv.messages` → 打包全部上下文统一发送
- 流式发送逻辑抽取为 `_doStreamSend(conv, apiMessages, aiMsg)`，供正常路径和队列路径复用
- `stopGenerating()` 清空队列，排队气泡随下次渲染消失

**结构变化总览**：

| 旧（v5） | 新（v6） |
|----------|----------|
| `let isGenerating = false` | `conv.isGenerating = false` |
| `let chatLocked = false` | `conv.locked = false` |
| `let abortController = null` | `conv.abortController = null` |
| — | `conv._queue = []` |
| `sendMessage` 内联流式逻辑 | 抽取为 `_doStreamSend()` |
| — | 新增 `_flushQueue()`、`_appendQueuedMessage()` |
| `destroy` 只 abort 一个 | 遍历 abort 所有对话 |
| `_loadFromStorage` 仅还原 | + 错误恢复 + 初始化缺失字段 |

**CSS 新增**（`components.css`）：
- `.queue-indicator` + `@keyframes pulse-dot`：排队气泡下方"排队中…"文字 + 呼吸圆点动画

### 聊天 — 用户气泡长文本溢出修复

**问题**：用户发送超长消息（URL、代码、无空格连续字符）时，内容超出气泡边界，甚至溢出整个对话区域。

**原因**：`.msg-bubble.ai-bubble` 有 `word-break: break-word`，但 `.msg-bubble.user-bubble` 没有。无空格的长字符串无法被浏览器自然断行，直接撑破 flex 容器。

**修复**（`components.css`）：`.msg-bubble.user-bubble` 新增两行：
```css
word-break: break-word;
overflow-wrap: break-word;
```

`word-break: break-word` 让浏览器在必要时从任意字符处断行（非 CJK 文本也能断），`overflow-wrap: break-word` 是兼容性兜底。

### 知识库 — 隐藏/删除按钮 Bug 修复

**Bug 1：删除按钮点一次弹出多个确认窗**

**原因**：`_bindEvents()` 每次 `_render()` 都执行 `el.addEventListener('click', ...)`，而 `_render()` 在 mount、切换视图、展开卡片、保存、删除等操作后都会被调用。第 N 次渲染后，DOM 上绑了 N 个 click 委托监听器。点一次"删除"，N 个监听器各触发一次 `_deleteItem()`，各创建一个 overlay，堆叠在一起。

**Bug 2：隐藏按钮"不生效"**

同样原因——N 个监听器同时触发 `_toggleVisibility()`。第一个调用 toggle 成功（如 visible→hidden），第二个紧接着又 toggle（hidden→visible），最终状态不变，看起来"按钮没反应"。

**修复**（3 处改动）：

| 改动 | 说明 |
|------|------|
| 新增 `_delegatedBound` 标志 | `mount()` 后首次 `_bindEvents()` 才绑定操作按钮的 click 委托，后续 `_render()` 跳过 |
| 操作按钮事件中加 `e.stopPropagation()` | 防止事件冒泡到卡片点击等其他处理器 |
| `_deleteItem` 加 `_deleting` 防重入 | 确认窗显示期间忽略后续调用；destroy 时重置标志 |
| 弹窗按钮改用 class 选择器 | `id="kDelConfirm"` → `class="k-del-confirm"`，防止同名 ID 冲突 |

**教训**：事件委托（在父容器上 `addEventListener`）只需绑定一次。如果绑定函数被多次调用（如每次 render），必须加标志防重复。

### 文件树 — 字体缩放时图标与文件名重叠修复

**问题**：用户在设置中调整字体大小时（`lubina_font_size` → `document.documentElement.style.fontSize`），文件树内的图标文字（如 "JS"、"C++"、"⬡"）和展开箭头 "▶" 溢出固定宽度的容器，与文件名重叠。

**原因**：`.tree-node-icon` 使用固定 `width: 16px`、`.tree-node-arrow` 使用固定 `width: 14px`，但它们的 `font-size` 使用 `rem` 单位随根字体缩放。大字体时图标文字溢出容器。`.tree-node` 的 `gap: 5px` 也是固定 px，不随字体缩放。

**修复**（`components.css`）：

| 选择器 | 旧值 | 新值 | 原理 |
|--------|------|------|------|
| `.tree-node` | `gap: 5px` | `gap: 0.3em` | em 相对父元素 font-size，随缩放等比变化 |
| `.tree-node-icon` | `width: 16px` | `min-width: 1.2em` | em 相对于自身 font-size: 0.9rem，等比缩放 |
| `.tree-node-arrow` | `width: 14px` | `min-width: 0.9em` | 同上，em 相对于自身 font-size: 0.65rem |

同时新增 `.tree-node-icon svg { width: 1em; height: 1em; vertical-align: middle; }` 让 SVG 文件夹图标也随字体等比缩放。

### 编辑器 — 多个 Bug 修复

1. **未修改文件误标"未保存"**：打开文件后不编辑直接切换，前一个文件被标为未保存。

**目的**：Agent（Plan/Auto 模式）将来操作文件时，只能碰用户选择的文件夹。
