### 文件树 — 字体缩放时图标与文件名重叠修复

**问题**：用户在设置中调整字体大小时（`lubina_font_size` → `document.documentElement.style.fontSize`），文件树内的图标文字（如 "JS"、"C++"、"⬡"）和展开箭头 "▶" 溢出固定宽度的容器，与文件名重叠。

**原因**：`.tree-node-icon` 使用固定 `width: 16px`、`.tree-node-arrow` 使用固定 `width: 14px`，但它们的 `font-size` 使用 `rem` 单位随根字体缩放。大字体时图标文字溢出容器。`.tree-node` 的 `gap: 5px` 也是固定 px，不随字体缩放。

**修复**（`components.css`）：

| 选择器 | 旧值 | 新值 | 原理 |
|--------|------|------|------|
| `.tree-node` | `gap: 5px` | `gap: 0.3em` | em 相对父元素 font-size，随缩放等比变化 |
| `.tree-node-icon` | `width: 16px` | `min-width: 1.2em` | em 相对于自身 font-size: 0.9rem，等比缩放 |
| `.tree-node-arrow` | `width: 14px` | `min-width: 0.9em` | 同上，em 相对于自身 font-size: 0.65rem |

同时新增 `.tree-node-icon svg { width: 1em; height: 1em; vertical-align: middle; }` 让 SVG 文件夹图标也随字体等比缩放。

### 聊天 — 发送拦截时不清空输入框

**问题**：当用户在 Plan/Auto 模式下未选择工作区就发送消息时，系统弹"需要工作区"提示并拦截发送，但输入框内容已被清空。用户恢复操作后需重新输入。

**原因**：输入框清空逻辑写在 `app.js` 的发送按钮/Enter 处理器中，紧接 `Chat.sendMessage(v)` 之后无条件执行。`sendMessage` 因工作区检查提前 return 时，清空已发生。

**修复**（3 处改动）：

| 文件 | 改动 | 说明 |
|------|------|------|
| `chat.js` → `sendMessage()` | 在 `_lock()` 前新增输入框清空逻辑 | 所有检查通过后才清空，此处是"确定要发送"的分界点 |
| `app.js` → 发送按钮 click | 移除 `inp.value = ''; inp.style.height = 'auto'` | 不再无条件清空，由 `sendMessage` 负责 |
| `app.js` → Enter 键 keydown | 同上 | 同上 |

将来添加余额检查等其他拦截逻辑时，也只需在 `sendMessage` 中提前 return（不清空输入框），无需修改 `app.js`。

### 文件树 — 新文件夹信任安全检查

**需求**：用户打开一个从未用 Lubina 打开过的文件夹时，弹出安全确认提示，确保用户信任该文件夹中的内容。

**实现**（`file-explorer.js`）：

### 编辑器 — 深色模式语法高亮重做（第二次修复）

**第一次尝试失败**：在 `EditorView.theme()` 里写了 20+ 个 `.cm-keyword`、`.cm-string` 等 CSS 选择器，但 CM6 的 DOM 上根本没有这些类名！语法着色走的是 tag→style 映射系统，不是 CSS 类名。用户反馈"完全没看到修改"。

**字体加粗**：深色模式下 `fontWeight: '450'`（`&` 和 `.cm-line`），行号和行号区保持 400。450 比默认 400 稍粗但不至于像 600 那样刺眼。

### 编辑器 — 多个 Bug 修复

1. **未修改文件误标"未保存"**：打开文件后不编辑直接切换，前一个文件被标为未保存。

   **原因**：两处问题——
   - `_saveCurrentContent()` 里比较 `cmView.state.doc.toString() !== f.content`，但 CM6 初始化时会规范化换行符或末尾空行，导致内容"看起来一样"但不相等
   - `updateListener` 的 `initial` 标志只跳过第一次 `docChanged`，CM6 初始化期间可能触发多次
   - **修复**：① `_saveCurrentContent()` 不再比较，直接无脑同步内容（`modified` 只由 updateListener 管理）；② 用 300ms `setTimeout` 设置 `_cmSettled` 标志，初始化期间的 `docChanged` 一律忽略

2. **Ctrl+C 复制不生效**：Ctrl+A/Z/X/V 有效但复制无效。

   **分析**：不是 JS 层面拦截的，可能是 Tauri 2 webview 对原生复制事件支持不稳定。CM6 的 `basicSetup` 自带 `standardKeymap` 包含 `Mod-c` 但可能没生效。
   - **修复**：显式添加 `keymap.of([{ key: 'Mod-c', run: ... }])` 键绑定，用 `navigator.clipboard.writeText()` + `execCommand('copy')` 兜底

### 文件树 — 真实文件系统 + 工作区安全

**目的**：Agent（Plan/Auto 模式）将来操作文件时，只能碰用户选择的文件夹。

**做了什么**：

1. **文件夹选择**：用 `tauri-plugin-dialog` 的 `open({ directory: true })` 弹出系统原生文件夹选择器。参数必须用 `{ options: { directory: true } }` 包装（Tauri v2 IPC 规范）。

2. **真实目录读取**：tauri-plugin-fs 的 `read_dir` 有 scope 限制（只让读应用自身目录），**解决方案**：在 `lib.rs` 写自定义 Rust 命令 `list_dir` + `read_file_content`，直接用 `std::fs` 读取，完全绕过插件 scope。
   - 前端通过 `window.__TAURI_INTERNALS__.invoke('list_dir', { path })` 调用
   - 返回 `Vec<DirEntry>`：`name` / `path` / `is_dir`（注意是 snake_case）
   - 按「目录优先 + 字母排序」

3. **懒加载**：目录节点标记 `lazy: true`，点击展开时才 `listDir` 读取该子目录内容。大目录（>200 条目）不会一次性全读。

4. **工作区安全校验**：
   - `file-explorer.js` 打开文件夹后设置 `window.__lubina_workspace_root`
   - `chat.js` 的 `sendMessage()` 中，Plan / Auto 模式**都**会检查：无工作区 → 弹"需要工作区"引导用户先选文件夹
   - `app.js` 提供 `isPathInWorkspace(path)`：防路径穿越（禁止 `..`）+ startsWith 判断
   - API 请求体传 `sandbox_root` 给后端

5. **bridge.js 全链路修复**（沿路遇到的所有坑）：
   - `window.__TAURI__` 不存在 → 项目未装 `@tauri-apps/api`，改用 `__TAURI_INTERNALS__.invoke()`
   - dialog 参数需要 `{ options: {...} }` 包装
   - fs 插件 scope 限制 → 自定义 Rust 命令绕过

   - 技术方案：Compartment 包装 `EditorView.lineWrapping` Facet，`reconfigure()` 动态切换

4. **Ctrl+S 保存**：CM6 `keymap.of([{ key: 'Mod-s', run: saveFile }])`，调用 `Bridge.writeTextFile()`

5. **未保存弹窗**：
   - 关闭文件标签（`closeFile`）：自定义 modal 替代 `window.confirm()`（Tauri v2 WebView 禁用原生 confirm）
   - 切换页面离开编辑器（`showPage`）：`_editorUnmountCheck()` 检查是否有 dirty 文件，弹"保存/不保存/取消"
   - 右键菜单新增"保存"选项
